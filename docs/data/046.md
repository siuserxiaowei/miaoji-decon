## 📚 深挖：4 个值得单独学的点

### 1. 一期功能清单是怎么被"砍"出来的
甲方一口气报了一串想要的功能：语音交互（支持打断）、声纹、语音唤醒、小夜灯、双目眼睛表情、摄像头、4G。乙方的工作不是全盘接，而是逐项判定：
- <strong>直接能做</strong>：语音交互+打断、声纹（"要不要都行"）、语音唤醒、小夜灯、双目眼睛表情——进一期。
- <strong>比较麻烦 → 放二期</strong>：摄像头"跟着人走"的交互。
- <strong>待评估</strong>：4G（服务端协议麻烦，要工程师"司机"评估）。
这就是硬件外包"范围谈判"的核心动作——把模糊的"我想要很多交互"收敛成"一个月能交付的确定清单"。<strong>你能用</strong>：自己提硬件/软件需求时，预期对方会逐项砍，提前想好哪些是"必须一期"、哪些可以让步。

### 2. "小夜灯"为什么是唤醒微光，不是智能家居控制
中间有个被现场澄清的误解：甲方说"小夜灯"，乙方一度以为是"控制你家里的灯/任务"那种智能家居场景——复杂度高、影响进度。澄清后确认其实只是——<strong>唤醒娃娃时亮一个比较暗的灯</strong>，让人眼适应，是个轻量交互。同一个词（"灯"），甲乙方脑中是两套复杂度完全不同的东西，差点把简单功能误判成进度风险。<strong>你能用</strong>：需求对接时，对每个名词都追一句"具体要什么效果"，别让一个词的歧义吃掉工期。

### 3. "眼睛"的具体效果：双目一致 + 表情包，且要敲定素材谁出
甲方对"眼睛"的要求说得比较细：<strong>双目</strong>、两只眼睛动作一致（"不会一个人这样一个人那样"）、带表情（眨眼、伤心难过等"眼睛的表情包"）。还顺带问了一个分工问题：<strong>表情包是甲方提供还是乙方做</strong>（结论是甲方最好有，乙方也可以想办法）。硬件交互里"眼睛"这种看似简单的视觉，落到工程上要明确表情资源谁出、双目同步怎么保证。<strong>你能用</strong>：定视觉/UI 类需求时，一并把"素材谁提供"敲定，否则后期扯皮。

### 4. 4G、双网切换与"不增加额外工作量"原则
甲方希望一期带 4G：逻辑是"有 WiFi 用 WiFi，没 WiFi 用 4G，像手机一样双网"。乙方提示这会让<strong>服务端协议变麻烦</strong>，需要工程师评估，没当场答应。关于 WiFi/4G 怎么切换，甲方明确表态：<strong>"你们现成怎么方便怎么来，用个按键就行，我不想给你们增加额外工作量、拖长工期。"</strong> 这体现了甲方一个清醒的取舍——<strong>为了保工期，主动放弃对实现细节的定制权</strong>。（另外有一处技术讨论：服务端用 Java 还是 Python——甲方倾向 Java 便于二次开发，乙方提示 AI 侧 Python 更成熟、Java 库少；双方未纠结，留到一期确认能做之后再谈。）<strong>你能用</strong>：当你是甲方，分清"必须我说了算"的和"交给乙方现成方案就行"的，后者越多工期越稳。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：用"一期能不能交付"做唯一裁决尺</strong>
- 核心意思：对每个功能只问一句"这一期能交付吗"，能就进、不能就往后排，不纠缠技术争论。
- 你可以怎么用：做需求评审时，先快速过一遍可行性，把范围定了再讨论实现细节。
- 不要误读成：把所有"麻烦的"都砍掉——是"往后排/转待评估"，不是放弃。

<strong>方法 2：对每个需求名词追问"具体要什么效果"</strong>
- 核心意思："灯""眼睛""摄像头"在甲乙方脑中复杂度可能差几倍，必须当场对齐。
- 你可以怎么用：听到一个功能词，先反问效果、再估工作量。
- 不要误读成：只要把词写进文档就算对齐了——歧义往往藏在词背后。

<strong>方法 3：把"现成方案"作为保工期的默认选项</strong>
- 核心意思：非关键细节（如双网切换方式）直接用乙方现成做法，不定制。
- 你可以怎么用：分清哪些必须定制、哪些可以"你们怎么方便怎么来"。
- 不要误读成：所有都听乙方的——只在不影响产品核心体验的细节上让渡。

- <strong>Day 1</strong>：把你手上项目的功能清单列出来，给每一项贴"能做一期 / 麻烦放后面 / 待评估"三选一标签。
- <strong>Day 2</strong>：挑清单里最像"小夜灯"那种容易被误解的功能，逐个追问"具体要什么效果"，写清楚。
