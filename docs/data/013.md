## ⏱ 如果只读 10 分钟（5 条带走判断）

1. <strong>研发和量产，一定拆成两份合同，别混在一份里。</strong> 全场最强共识。混在一起会"扯来扯去拉个半年"，量产遥遥无期，团队垫了钱却收不回尾款。拆开后：样机验收完就结清研发款，量产是另起一份合同的事。
2. <strong>量产合同充满不确定性，不该现在就锁死。</strong> 市场推广做不做得起来、产品到底能不能卖出去、要不要继续研发——都没定。"太不确定的东西，不要现在就定。"先把能交付、能验收、能结款的研发部分签掉。
3. <strong>付款比例的设计，本质是在对冲"对方拖延"的风险。</strong> 团队在 532 / 4411 / 30% 几个方案间反复拉扯，核心焦虑是：前期要垫钱（物料、打印、人力都要钱），如果尾款绑在"对方量产"上，对方一拖半年，钱就拿不回来。

## 📚 深挖：5 个值得单独学的知识点

### 1. 研发/量产拆两份合同——长周期外包的"风险断点"设计
现场反复强调："这个量产太不确定了"，"混在一起到时候扯来扯去搞个半年"，"遥遥无期"。设计是：<strong>第一份研发合同</strong>金额小（讨论中从 5 万 / 7 万 / 10 万反复横跳），做出几台样机，甲方体验、验收功能，<strong>验收完就结清全款</strong>，并交付硬件物料清单（BOM）、软件文档、组装 SOP（"作业指导书"）。<strong>第二份量产合同</strong>才去找模具厂评估开模费、谈阶梯价（500/1000/5000~10K 套不同单价）。
<strong>为什么重要</strong>：把一个"高不确定、长周期"的项目，切在"我方可独立交付且可验收"的天然断点上。研发是乙方能控制的，量产依赖甲方的市场、资金、决心——把这两类风险分开承接，是所有长周期乙方都该有的本能。
<strong>你能用</strong>：接外包时，先找到"我交付完、对方无法赖账"的那个节点，把合同切在那里。

### 2. 付款比例（532 / 4411）不是面子，是现金流防线
团队在几个方案间拉扯：原合同是 <strong>532</strong>（分三阶段：做好 50%？→ 视频/样机做好 30% → 确认无误 20%，逐字稿对具体阶段描述有冲突，存疑）；如果研发量产<strong>不拆</strong>，就要把比例提到 <strong>4411</strong>（首付更高），因为"前期我还得投钱进去搞"。还有一条线是对甲方报的是 <strong>30%</strong>（"你已经给他看过 30% 了，那就 30%"）。
<strong>为什么重要</strong>：这串数字背后只有一个逻辑——<strong>前期全是垫钱</strong>（"物料要钱、打印要钱、人力要钱"），所以要么把首付比例拉高，要么把结款节点前移到样机验收。比例之争本质是"我的钱什么时候能安全落袋"。
<strong>你能用</strong>：设计付款节奏时，把现金流缺口最大的阶段对应到最高的收款比例；别让大额尾款悬在你不可控的事件上。⚠️ 532/4411 具体对应哪三/四个阶段，逐字稿说法不一，落地前需以实际合同为准。

### 3. 给甲方"不二选一" + 埋达量返点钩子
有人主张给甲方两个方案选（签两份 vs 签一份调高比例），被明确否掉："不用他选，直接分两份合同。""那你还写 41 没意义了"——既然不让选，就别把谈判筹码都摊开。但同时在量产合同里留了软钩子："如果你真要做量产、能达到这个量，我就返给你""他量产从我们拿货，我们就抵扣货款（提到 8%）"。
<strong>为什么重要</strong>：这是一手"减少对方决策维度、同时用利益绑定降低对方违约动机"的组合。不给选项=降低博弈成本；达量返点=把对方"拿了资料跑去别处量产"的风险，用利益对冲掉一部分。
<strong>你能用</strong>：谈判中能拍板的别给选项；对方有"拿了东西跑路"风险时，用"达标返利/从我拿货抵扣"绑住他。⚠️ 团队自己也承认这钩子约束力有限——"你也限制不了他拿你资料去别的地方"，第一阶段资料全交付后，对方完全可以自己找模具厂。这是结构性死穴，未解。

### 4. 5G WiFi 选型：ESP32-P4 + C6，而不是 S3 外挂
甲方要 5GHz WiFi。讨论结论：ESP32-S3 本身不支持 5G，要外挂第二个 WiFi 模块，"两个 WiFi 容易互相干扰"，量产 2 万台很麻烦。而 <strong>ESP32-P4 + C6</strong> 协处理器<strong>原生支持 5GHz WiFi</strong>，"只贵四五块钱""贵不了多少"。（逐字稿把型号识别得很乱——"T4/P4""C5/C6""一六/16"混杂，但意思清晰：主控 P4 + 带 5G WiFi 的协处理器。C6 实际只支持 WiFi 6 的 2.4G，团队中有争议，最终倾向 P4+C6 方案。）另提到轮子要用<strong>带反馈的电机</strong>（360°旋转 + 反馈控制），是个移动型设备（疑似 AI 陪伴/玩具机器人）。
<strong>为什么重要</strong>：典型的"多花几块钱料钱，省掉一整类量产麻烦（干扰、调试、外挂模块）"的工程决策。在量产 2 万台的规模下，单台几块钱 vs 一类系统性风险，账很好算。
<strong>你能用</strong>：硬件选型时，把"BOM 多几块钱"和"它能消灭哪一类量产期的系统性麻烦"放一起算总账，别只盯单台成本。

### 5. 视觉识别：买百度 API（含 license）vs 自建 function call 调 VL 模型
研发侧有人提议：拍照识别能不能不接百度的视觉接口，自己做个 function call 调一个 VL（视觉语言）模型，"是不是很省钱？"被否："你算一下成本，别搞几天就比 license 还贵。"关键算账：百度方案 license 加了 15 块钱（疑为单台/单授权口径），但"如果一直拍照，图片 token 量大，15 块用不了多久"。后来反转——有人说"我们现在用百度的，百度给的 license 钱出来后就可以一直用，不用管 token"，于是"使劲用、天天发图、一秒一张"，因为"那么多大厂机器闲着也是闲着"。还讨论了<strong>压缩图片提高压缩比来省 token</strong>，但若走百度买断 license 就"不关我们的事"。
<strong>为什么重要</strong>：这是一个"自建 vs 采购"的经典成本权衡，且关键变量是<strong>调用量</strong>——高频拍照场景下，按量付费（token）会迅速吃掉看似便宜的自建方案，而买断/license 在高频下反而划算。
<strong>你能用</strong>：评估 AI 能力是自建还是买，先估<strong>真实调用频次</strong>；高频场景优先看有没有买断/包月，按 token 计费在高频下经常是陷阱。⚠️ "15 块""一直用"这些都是现场口径，具体计费方式需查百度实际定价。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：把长周期项目切在"可独立交付且可验收"的断点上分段签</strong>
- 核心意思：研发（你可控）和量产（依赖对方）分两份合同，样机验收即结研发款。
- 你可以怎么用：接外包先找那个"交付完对方赖不掉"的节点，合同切在那里。
- 不要误读成：所有项目都拆两半——只有当后半段不确定性高、且前半段能独立验收时才拆。

<strong>方法 2：付款比例跟着现金流风险走，不跟着面子走</strong>
- 核心意思：垫钱最多的阶段，对应最高的收款比例 / 最早的结款节点。
- 你可以怎么用：算清每阶段你垫多少，把高比例压在你能交付验收的节点上。
- 不要误读成：一味要高首付——对方也会压价，要的是"风险匹配"，不是单纯多收。

<strong>方法 3：自建 AI 能力前先估调用频次再算账</strong>
- 核心意思：高频调用下，按 token 计费会迅速超过看似便宜的买断/license。
- 你可以怎么用：先估真实日调用量，再比"自建 function call"和"买断 license"的总成本。
- 不要误读成："自建一定省钱"——低频才省，高频常常更贵。
