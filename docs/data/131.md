> 场景：一通甲乙双方的需求洽谈电话。甲方是某央企的新员工（自称没嵌入式基础、之前没接触过这块），手里有一个买好的微雪（Waveshare）智能手环硬件，要乙方（一家能开技术服务费发票的软件开发公司）帮忙做手环固件开发；乙方在套需求、谈打款、约报价时间。
>
>
> ⚠️ 这不是一场"知识分享会"，没有大佬讲干货。它的学习价值在另一面：<strong>怎么从一个说不清需求的客户嘴里把项目边界、付款方式、政企采购的暗礁问出来</strong>——一份外包/政企单子洽谈的真实样本。

## ⏱ 如果只读 10 分钟（6 条带走判断）

1. <strong>这是一单"央企外包 + 个人对接"的硬件固件开发</strong>：手环已买（微雪，ESP32-S3 芯片），客户要的是固件——抬手唤醒、采音 5~10 秒、WiFi 打包发本机、收结果后屏显"好/坏"。模型和服务端是甲方自己做，乙方只负责硬件这一端把"能直接调用的 WAV 音频"发出去。
2. <strong>客户技术上是真小白，需求会变</strong>：对接人是央企新员工，靠 ChatGPT 改示例代码已经跑通了 WiFi 和陀螺仪唤醒屏幕，但卡在麦克风采样（每次 1024 点）合成 WAV 播放不了、LVGL9 例子太少 AI 写不出来。他明说领导（一个"博士"）也没想清路线，<strong>签完合同后大概率还会改需求</strong>。
3. <strong>政企采购的三道暗礁全暴露了</strong>：① 必须走对公转账、不能走个人；② 定金公司出不了、要对接人<strong>个人垫付</strong>；③ 没有淘宝对公店铺就没法走"项目一开始全款打款"那条路。这三条决定了乙方能不能拿到钱、什么时候拿到。
4. <strong>发票口径是成交前提</strong>：客户先问"能不能开票"才往下谈。最终敲定开<strong>技术服务费</strong>普票（理论上要专票，但普票也收，专票就把税费记进总额）。乙方"有几家技术有限公司"能开票，是这单能成的硬条件。
5. <strong>客户的真实诉求是"细水长流地交付"</strong>：对接人反复说"做得越慢越好""我也想摸鱼""每周都能汇报一点点"，年底交一部分功能即可。这不是玩笑——他要的是<strong>可持续向领导汇报的进度</strong>，乙方接住了（"我们也做过 G 端，每周写周报"）。
6. <strong>乙方的标准动作很专业</strong>：先评估再报价、本周内（周四）出方案、要硬件型号链接、把"WiFi 协议设计/多设备寻址"这些客户没想到的复杂点主动点出来并计入工作量——把一个模糊需求收敛成可报价的方案。

## 🎯 这篇真正能学什么

这场会练的不是"AI 硬件技术"，而是<strong>乙方视角的需求洽谈眼力</strong>。读完该记住三张图：① 怎么把一个技术小白客户的口语化描述，翻译成"硬件端做什么、服务端做什么、边界在哪"的可报价方案；② 政企/G 端单子在"钱"上的特殊结构（对公、垫付、发票、淘宝对公）——技术再好，卡在打款就白搭；③ 怎么读懂客户没明说的潜台词（要慢、要能汇报、需求会变），并把它变成对自己有利的合作节奏。别记手环参数，记这三张谈判与交付的图。

## 📚 深挖：5 个值得单独学的知识点

### 1. 把"小白客户的口语需求"翻译成可报价的技术边界
现场客户的描述是高度口语化的："带上手环—抬手—唤醒屏幕—麦克风采 5~10 秒音频—WiFi 发本机/电脑—跑模型判好坏—结果回传屏显好/坏，并烧个音频用扬声器播报。"乙方在听的过程中不断做<strong>边界切割</strong>：模型和服务端归甲方、硬件固件归乙方、乙方只需保证"发出去的是能直接调用的 WAV"。
<strong>为什么重要</strong>：外包纠纷大多不是技术做不出来，是边界没切清——"这个算谁的"。把"输入是什么、输出是什么、交付物是什么"在洽谈阶段就明确，是报价和验收的地基。
<strong>你能用</strong>：接任何外包，第一件事是画一条"你做什么 / 我做什么 / 交接面是什么格式"的分界线，并让客户口头确认。

### 2. 政企/央企采购在"钱"上的三道坎
这场会把政企单子的资金暗礁讲得很透：① <strong>必须对公转账</strong>，走个人不行；② <strong>定金公司出不了</strong>，得对接人个人垫付（对接人甚至说"我倒愿意垫"）；③ 想让"项目一开始就全款打款"，乙方得有<strong>淘宝对公店铺</strong>这类可对公收款的通道，否则走不通。客户还主动说定金一般"五个或四六开（50% 或 40/60）"。
<strong>为什么重要</strong>：央企/G 端的钱安全但流程死板。技术能力是入场券，<strong>合规收款能力</strong>才是能不能成单、什么时候回款的决定因素。
<strong>你能用</strong>：服务政企前，先备齐"对公账户 + 能开技术服务费/专票的主体 + 一个可对公收款的电商通道"，把回款路径打通再谈技术。

### 3. 发票口径：技术服务费 / 普票 vs 专票
客户开场第一句就是"你们能不能开发票"。后续敲定：理论上要专票，普票也能收；开<strong>普票</strong>时若客户要专票，就把税费记进总额（"金额贵一点没什么"）；明细开<strong>技术服务费</strong>。乙方答"我们有几家技术有限公司，可以开"。
<strong>为什么重要</strong>：能不能开票、开什么票、税费谁承担，直接决定报价含不含税、客户能不能内部报销。这是 toB/toG 报价单里必须前置确认的一栏。
<strong>你能用</strong>：报价前先确认三件事——开普票还是专票、明细写什么（技术服务费/开发费）、税点谁承担，再给最终价。

### 4. 读懂客户的潜台词："做得越慢越好"
对接人反复表达：希望乙方"做得越慢越好""我也想摸鱼""不想这么快做完领导就叫我推下一步""每周都能汇报一点点"。表面像玩笑，实质是新员工的<strong>真实工作诉求</strong>——他要的是一个能持续向领导（那位"博士"）交差、显得每周都在干活的项目节奏。乙方立刻接住："我们也做过 G 端，每周写周报。"
<strong>为什么重要</strong>：对接人的个人 KPI（向上汇报、有存在感）往往和公司的项目目标不完全一致。能服务好"对接人这个人"，单子才稳。
<strong>你能用</strong>：谈单时分清"公司要什么"和"对接人个人要什么"，用交付节奏/周报/阶段性可见成果去满足后者，他会成为你单子内部的盟友。

### 5. 主动暴露客户没想到的复杂点（多设备寻址 / 通信协议）
客户天真地以为"发两个字符给开发板，板子显示好/坏就行了"。乙方点破：如果不止一个设备，"我怎么知道你发给谁？肯定要有一套协议"——WiFi 只是联网方式，上层还得有 TCP/MQTT/Socket 这类协议来做设备寻址和消息分发。客户当场承认"我对这个没有类似开发经验"，并接受把"协议设计"计入工作量、提前帮做。
<strong>为什么重要</strong>：主动揭示隐藏复杂度，既建立了专业信任，又把工作量（=报价）做大，还避免了后期"这个你怎么没做"的扯皮。这是乙方把模糊需求转成更大、更稳订单的关键动作。
<strong>你能用</strong>：评估需求时，主动列出客户没意识到的工程复杂点（多设备、协议、并发、容错），写进方案并计价——专业感和议价权一起到手。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：先评估再报价，绝不当场报价</strong>
- 核心意思：客户问"收费怎么样"，乙方答"先去评估"。把评估、出方案、报价拆成本周内交付的动作（周四给方案），而不是被逼着拍脑袋报数。
- 你可以怎么用：面对模糊需求，统一话术"先评估、出方案再报价"，给自己留出把边界和工作量算清的时间。
- 不要误读成：拖延。乙方明确承诺了时间点（周四），既稳又快，是"专业延迟"不是"敷衍"。

<strong>方法 2：分阶段签约 + 变更走追加协议</strong>
- 核心意思：第一版先做核心功能，后续优化"如果工作量大就重新签一个协议、再打一笔钱"。把"客户会改需求"这个风险，提前用合同结构化解。
- 你可以怎么用：合同里写清"本期范围 = X；超出范围的变更另行计价、追加协议"，把变更从纠纷变成增量收入。
- 不要误读成：每次改都狮子大开口。小调整可计入当期工作量（现场乙方就同意把"端侧判断声音结束"记进第一期），大变更才追加——尺度要让客户觉得公道。

<strong>方法 3：用客户的口去确认交接格式</strong>
- 核心意思：乙方反复确认"你们要什么格式？一定是 WAV 吗？""结果是你服务器下发的吗？""硬件和服务器的通信格式到时给我"。把交接面的每个接口都问到客户亲口确认。
- 你可以怎么用：凡是双方各做一半的项目，把"我给你什么格式、你给我什么格式"逐条问清并落到方案里。
- 不要误读成：客户说了就一定对。客户连"服务器还没定"都说了，所以接口要约定"可适配/待定"，别把未定的东西当确定的写死。
