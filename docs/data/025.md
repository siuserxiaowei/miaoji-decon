1. <strong>需求方常常说不清自己要什么，对接的活儿是"帮他把需求逼出来"</strong>。客户原话"你什么都问我，没体现你的专业性"，本质是希望乙方主动给方案、给选项，而不是被动等需求文档。

3. <strong>通信双通道：单板用网口直出，多板级联用 CAN 串起来再从一个节点网口出去</strong>。一个现场可能 20~30 个节点，不想拉几十根网线，用 CAN 串联、UDP 汇总出口。
4. <strong>协议不必标准化，自定义一个结构体打包发送就够</strong>——但要算通信压力。客户倾向"整板状态打包成大结构体发"，工程师提醒：几十台同时这么发压力很大，键值（ID+状态）增量发送是更省带宽的另一选项。
5. <strong>大电流负载（点很多灯）必须单独考虑供电</strong>。工程师主动追问"同时点 50 多个灯是不是要单独供电"——单片机 IO 直驱电流不够，这是新手最容易漏的硬件细节。
6. <strong>没文档的需求会，记录责任要当场认领</strong>。客户没有任何文字资料，工程师当场认领"我这边有记录，到时候发群里"——口头会必须有人收口成文，否则后面全是返工。

## 📚 深挖：6 个值得单独学的知识点

### 1. DI/DO/AI 是采集卡的三类基本信号
- <strong>是什么</strong>：客户要的核心功能就三类。DI（数字输入）=开关、按钮、旋钮，进来是高低电平（0 和 3.3V）；DO（数字输出）=去点亮 LED，输出高低电平；AI（模拟输入）=采集连续模拟量（如电位器电压、传感器读数），本场要 16 路。
- <strong>为什么重要</strong>：几乎所有"采集+控制"类硬件需求都能拆成"输入几路开关量 + 输出几路控制信号 + 采集几路模拟量"。把它当成三个抽屉，再复杂的板子也能先归类。
- <strong>你能用</strong>：接采集/控制类需求，先问清 DI 多少路、DO 多少路、AI 多少路，需求就清晰一大半。

### 2. "最大覆盖"统一设计：用一块板盖住所有规格
- <strong>是什么</strong>：客户有很多块面板、规格不一。最后定调"我按最大那块设计——64 路 DI、48 路 DO、16 路 AI，全用同一块板"，不再为每种规格单独开板。
- <strong>为什么重要</strong>：这是工程上的"标准化 vs 定制化"权衡。统一成最大规格，BOM、固件、库存、维护全收敛成一种，代价是小场景下有冗余 IO。多数情况维护成本远高于料件冗余成本，所以"做大盖小"更划算。
- <strong>你能用</strong>：当一个东西要量产/复用多次，优先评估能不能用一个最大规格统一，而不是按每个场景定制。

### 3. 通信分级：单板网口直出，多板 CAN 级联
- <strong>是什么</strong>：单独用一块板时直接走网口（UDP/以太网）出去；多块板级联时（一个现场 20~30 个节点），用 CAN 总线串起来，最后从某一个节点的网口统一出去。客户原话："不想用这么多网线，所以用 CAN 串起来，用一个网络。"工程师建议做两种版本（带网口/不带网口），但客户希望每块板都带网口，灵活度最高。
- <strong>为什么重要</strong>：这是多节点系统的经典布线权衡。CAN 适合"一根线串多节点、抗干扰、距离远"，但带宽有限；以太网带宽大但布线成本高。级联用 CAN、出口用网口，是两者取长补短。
- <strong>你能用</strong>：设计多节点系统时先问"节点之间怎么连、整体怎么对外"，CAN 做内部总线 + 网口做外部出口是成熟范式。

### 4. 协议自定义：结构体打包 vs 键值增量，两种发法各有代价
- <strong>是什么</strong>：客户不要标准协议，"自定义一个结构体，把整块板的状态打包发"就行。工程师提出另一种发法——键值方式（一个 ID 码带一个状态，按一下发一下，增量上报）。客户倾向打包发，但自己也担心"通信压力会不会很大"。工程师直言："那肯定大了，你一下子那么多"，尤其几十台同时发。
- <strong>为什么重要</strong>：这是数据上报的核心权衡。<strong>整包发</strong>简单、状态完整，但每次发全量，节点一多就堵带宽；<strong>键值增量发</strong>只发变化量，省带宽，但实现稍复杂、需处理丢包同步。选哪种取决于节点数量和实时性要求。
- <strong>你能用</strong>：设计上报协议别只图省事整包发，先估"节点数 × 包大小 × 频率"会不会压垮总线/网络；量大就改增量上报。

### 5. 大电流负载必须单独供电——单片机 IO 直驱带不动
- <strong>是什么</strong>：工程师主动追问："如果同时点 50 多个灯，是不是得单独供电？单片机直接去驱动，电流可能不够。"客户最大一块板有 48 路灯驱动信号，每路串 4 个 LED。
- <strong>为什么重要</strong>：单片机 GPIO 总输出电流有上限（通常几十到上百 mA），点几十上百个 LED 远超限度，直驱会烧芯片或灯不亮。正确做法是用三极管/MOS/专用 LED 驱动芯片配独立电源。这是嵌入式新手最常翻车的点。
- <strong>你能用</strong>：任何"用 MCU 直接驱动大功率/多路负载"的设计，先算总电流，超限就加驱动电路和独立供电。

### 6. 需求澄清本身是一种专业能力——别只当传声筒
- <strong>是什么</strong>：会中客户两次表达不满——"你什么都问我，没体现你的专业性，我要什么都会我就自己做了""我搞不清楚你想了解什么"。而工程师确实在反问背景、用途、协议形式。
- <strong>为什么重要</strong>：甲方常常不知道自己要什么，期待的不是被动问卷式追问，而是乙方<strong>基于专业先给方案选项</strong>（如"通信你可以选键值或整包两种，区别是…"），再让甲方做选择题而非填空题。前者显专业、推进快；后者显业余、对方烦躁。
- <strong>你能用</strong>：接需求时，把"你想要什么？"换成"基于你的场景，我建议 A 或 B，区别是 X，你倾向哪个？"——用选项代替空白追问。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：需求收敛用"选项题"而非"填空题"</strong>
- 核心意思：甲方说不清需求时，乙方给 2~3 个带利弊的方案让对方选。
- 你可以怎么用：把"通信怎么发？"改成"键值增量 or 整包发，前者省带宽后者简单，你选？"
- 不要误读成：不是不问问题，而是别让甲方从零描述——给他选择题。

<strong>方法 2：能统一规格就别做多型号（最大覆盖）</strong>
- 核心意思：用一块最大规格的板盖住所有场景，收敛维护成本。
- 你可以怎么用：盘点要做的多个版本，看能否合并成一个最大规格 + 软件裁剪。
- 不要误读成：不是无脑堆料——要评估冗余料件成本是否真的低于维护多型号的成本。

<strong>方法 3：口头需求会必须当场收口成文</strong>
- 核心意思：没有文档的会，必须有人认领把要点整理成文字发出来。
- 你可以怎么用：会议结束前明确"谁记、发到哪"，本场是工程师认领"发群里"。
- 不要误读成：不是会后随便补——关键参数（路数、协议、供电）当场确认再落字。
