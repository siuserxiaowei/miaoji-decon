## ⏱ 如果只读 10 分钟（5 条带走判断）

1. <strong>甲方要的东西，本质是"私有知识库 + 角色设定 + 长记忆"的语音设备</strong>——不是新概念，是把 RAG（检索增强）塞进一个硬件外壳。听懂这句，就听懂了整通电话。
2. <strong>"本地化部署一套知识库"是方案商的核心卖点</strong>：甲方给数据，方案商在甲方自己服务器上部署，硬件只通过 API 接口连过去。硬件和大脑是分开的两层。
3. <strong>硬件做不到的事要当场说清</strong>：方案商明确否了"设备自己识别人脸/自己改数据库"——人脸识别要调云端 API 比对，写库要人工。<strong>一个靠谱的乙方，是会当场告诉你哪些做不到的那个。</strong>
4. <strong>"用什么平台搭"决定响应速度</strong>：用 Coze/Dify/n8n 这类工作流编排，好处是快速搭、看得见流程；代价是"走工作流肯定比小智慢"。低延迟语音要本地化部署 + 堆显卡（小智用了好几套 4090）。
5. <strong>长记忆 = 存文本不存语音</strong>：把语音转成文本再存，成本低、量小；存原始语音"成本就高了"。这是所有"AI 记住你"功能的通用工程底线。

这通电话练的不是"谈了个大生意"，而是<strong>当一个非技术甲方第一次找硬件方案商时，双方在对什么、卡在哪</strong>。读完该记住三张图：①一台"会聊天、有知识、记得住人"的 AI 硬件，在工程上被拆成哪几块（知识库 / 角色设定 / 长记忆 / 声纹人脸 / 算力）；②哪些是"调个 API 就能做"，哪些是"硬件做不到"；③一个甲方该怎么把模糊需求推进成可报价的方案（写需求清单 → 买样机测 → 拉群分工）。别记"谁要买几台"，记这三张拆解图。

## 📚 深挖：4 个值得单独学的知识点

### 1. "私有知识库语音设备"在工程上到底是哪几块
甲方的需求听起来像一个东西，其实方案商在电话里把它拆成了五块：
- <strong>知识库</strong>：甲方给数据，本地化部署；对话时设备"直接用你那一套里面的东西去回答"——这就是 RAG（检索增强生成），让模型只答你自己库里的内容。
- <strong>角色/模块设定</strong>：甲方想要"婚姻模块讲婚姻、小孩模块讲小孩"——方案商纠正这叫<strong>角色设定</strong>，不是知识库本身。术语上要分清：知识库是"答案来源"，角色设定是"用什么人设/范围来答"。
- <strong>长记忆</strong>：记住对话、跨次保存。
- <strong>声纹 + 人脸</strong>：记住"这个声音/这张脸是谁"。
- <strong>算力</strong>：本地化部署要堆显卡。

<strong>为什么重要</strong>：很多人把"AI 硬件"当成一个黑盒，谈需求时鸡同鸭讲。把它拆成这五块，甲乙双方才对得上话。<strong>你能用</strong>：下次描述一个 AI 产品需求，先按"数据来源/人设/记忆/识别/算力"五块各写一句，需求立刻清晰。

### 2. 硬件做不到什么——乙方当场划的红线
甲方反复想要"设备自己识别人脸、自己往数据库里写'这是张阿姨'"。方案商明确否了两点：
- <strong>设备自己改数据库</strong>：不行，写库要人工补写。
- <strong>设备本地识别人脸</strong>："硬件做不到这种程度"，要把图片传到云端调<strong>人脸识别 API</strong>比对，再返回"这是谁"。声纹同理。

<strong>为什么重要</strong>：这是判断乙方靠不靠谱的关键信号——<strong>会主动说"这个做不到/要走 API"的，比什么都答应的更可信</strong>。把识别能力放云端 API 而不是端侧，也是当前多数低成本硬件的现实架构。<strong>你能用</strong>：对接硬件方案时，专门追问"哪些是端侧做、哪些要调云端 API"，含糊其辞的要警惕。

### 3. 工作流编排平台的"快"与"慢"
甲方问"用什么平台搭最上手"，乙方答 Coze（扣子）/Dify/n8n 都行，因为它们是<strong>可视化工作流</strong>——把"识别→查库→生成→回答"画成流程，搭起来快、看得见。但代价被点破了："走工作流肯定比小智慢。"小智（开源语音助手项目）为了低延迟是本地化部署 + 堆算力（"用了好几套 4090"）。

<strong>为什么重要</strong>：这是一个通用权衡——<strong>编排平台（快速搭建/可维护）vs 原生本地部署（低延迟/高算力成本）</strong>。语音类产品对延迟极敏感，用户等 2 秒就觉得"傻"。<strong>你能用</strong>：做语音/实时交互，先定"能接受多少延迟"，再倒推该用工作流还是本地部署；做非实时（文档问答类）则工作流性价比更高。

### 4. 长记忆的工程底线：存文本，不存语音
甲方纠结"长期保存对话"怎么做。乙方点破：<strong>保存文本而不是语音</strong>——把语音转成文本再存，成本低、数据量小；"你不可能保存语音的"，存原始音频"那个就大了"。还提到可以<strong>存对话总结而非逐句</strong>，存储量更省。

<strong>为什么重要</strong>：这是所有"AI 记住你"功能背后的真实取舍——记忆不是录音，是结构化的文本/摘要。用户越多，这条成本曲线越陡。<strong>你能用</strong>：设计任何带"记忆/历史"的产品，默认存文本摘要而非原始媒体；并想清楚"全量存 vs 只存摘要"对成本的影响。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：把模糊 AI 需求拆成"五块清单"再谈</strong>
- 核心意思：数据来源 / 人设范围 / 记忆 / 识别 / 算力，五块各写一句。
- 你可以怎么用：下次对接外包或描述产品，先填这五块，再开口谈。
- 不要误读成：把"知识库"和"角色设定"混为一谈——前者是答案来源，后者是回答的人设/范围。

<strong>方法 2：先要"需求标准/报价"，再谈钱</strong>
- 核心意思：让乙方先写需求标准、出方案报价，把范围钉死。
- 你可以怎么用：对接时明确说"先给我一份需求清单+报价，我们再推进"。
- 不要误读成：上来就问"多少钱"——没钉清范围的报价没意义。

<strong>方法 3：用样机实测代替空想</strong>
- 核心意思：与其反复口头确认，不如买几台现有样机先测，看实际功能。
- 你可以怎么用：任何硬件/产品对接，尽早拿到可摸的样品测真实体验。
- 不要误读成：样机功能 = 最终定制功能——样机只代表"现有能力下限"，定制还要单独评估。
