> 场景：一通约半小时的技术售前电话。客户（带"强哥"等多人）想做一个工业场景下的 AI 语音交互硬件——用类似"小智"的语音设备，通过说话调出报表/图表/网页，显示在现场的大触摸屏电视上；供应商（胡总一方）反复追问，最后把这个模糊需求拆成"语音驱动电脑上的脚本/MCP"。
>
>
> ⚠️ 本场信息密度偏低，且高度重复：大半篇幅是同一个需求来回绕圈、互相听不清。真正值钱的不是"结论"，而是<strong>看一场需求澄清是怎么从一团浆糊被一点点逼清楚的</strong>。本篇按这个角度榨干，不硬凑商业大道理。

## ⏱ 如果只读 10 分钟（5 条带走判断）

1. <strong>客户说"我要 A"，真实想要的往往是 B</strong>。客户一直说"把小智连到电视上""投屏"，供应商追问到最后发现：根本不需要物理连接电视，要的是"语音驱动电脑/大屏上的软件去打开报表"——是个控制问题，不是连接问题。
2. <strong>供应商全程在做一件事：把模糊需求逼成可执行的技术规格</strong>。他反复问同一个问题的不同版本——"到底要干什么用？""最终传给硬件的是什么？""有几种类型？"——这就是售前最核心的功夫。
3. <strong>这个需求的技术答案是 MCP</strong>。供应商一句话点破：电脑当 MCP server，小智设备当 client 调用电脑上的能力（打开软件/网址/报表）。硬件本身只是语音入口，真正干活的是大模型 + 电脑本地能力。
4. <strong>客户怕被"脚本化"锁死，想要的是"可配置的灵活"</strong>。客户明确拒绝"写死的脚本"——"我今天打开这个、明天打开那个""我要灵活地去配置"。这是脚本（固定）和大模型驱动（动态）的本质分歧，也是这单的真正卡点。
5. <strong>离线/本地化是工业场景绕不开的暗礁</strong>。聊到一半冒出"某些场景没网络"，供应商立刻指出：没网大模型就调不了，除非全本地化部署（同一局域网内可调）。这是工业落地必须先问的一句话。

## 📚 深挖：5 个值得单独学的知识点

### 1. 客户的"投屏"其实是"控制"——需求澄清的经典误判
客户开场反复说"把小智连到电视上""投屏""HDMI 接口"。供应商顺着这个聊了很久（投屏就是 HDMI 把小屏原样扩展到大屏，分辨率不会变高），但越聊越不对，最后供应商一句话点破："那就不需要连屏幕了……你这个就相当于去直接控制一下设备/控制电脑上的一些东西。"
<strong>为什么重要</strong>：客户给的是"解决方案语言"（投屏），不是"问题语言"（我要语音调出报表显示在大屏上）。顺着客户的方案走，会做错东西。<strong>你能用</strong>：客户一上来给方案/给名词（"我要个投屏""我要个 App"），先翻译回"你要解决的问题是什么"，再判断方案对不对。

### 2. MCP：让语音盒子去调电脑能力的那层协议
供应商的核心技术判断：硬件不需要和电视物理连接，只要都联网（WiFi/4G/局域网）就能通信。架构是——"你在电脑上弄一个 MCP 的协议，电脑就是 MCP 的 server，小智那些东西作为客户端去调用上面的能力"；"小智只是语音入口，最后还是大模型去调用你电脑上本身那些能力，相当于 MCP"。
<strong>为什么重要</strong>：这是"AI 硬件控制现有系统"的通用解法——硬件只管把语音变成指令，真正的执行能力挂在电脑/服务端，用 MCP 暴露出来。换任何语音设备都行（供应商原话："这个设备你用任何一个设备都可以"）。<strong>你能用</strong>：想让 AI 设备控制你现有的软件/系统，思路是把系统能力包成 MCP server，设备/大模型作为 client 调用，而不是给每个设备做定制硬连接。

### 3. 脚本 vs 大模型驱动：客户真正的卡点在"灵活可配"
供应商最初的方案是"写个自动化脚本，语音转成调用脚本，把软件/浏览器打开"。客户明确反弹了好几轮："我们其实不想用那种脚本调用，太直接了""我今天打开这个、明天打开那个""我要灵活地去配置……不然每次都要去调整"。客户要的是"我可以自己去配网址/参数，通过语音灵活完成"，而不是被一段死脚本锁住。供应商也给出折中："关键字识别，预设一些模板，直接打开对应报表"。
<strong>为什么重要</strong>：这是"固定自动化（脚本，便宜但僵）"和"动态智能（大模型驱动，灵活但要工程）"之间的经典取舍，也是这单能不能成的真正分歧点——双方其实是在谈"灵活度交付到什么程度"。<strong>你能用</strong>：接 AI 自动化需求时，先和客户对齐"要写死的固定流程，还是要能自助配置/动态调整"——这决定了报价和工程量，含糊过去后面必返工。

### 4. 离线/本地化：工业场景的隐藏硬约束
对话中段客户突然提："某些场景下没有网络……我单独跟它在那边问它。"供应商立刻判断："没有网络，你大模型也调不了。"接着澄清 4G/WiFi 归根结底都是联网；真要断网，只能"全部本地化部署"，且"只要在同一个局域网里面就都可以调用"。
<strong>为什么重要</strong>：工业现场常有断网/内网隔离要求，而云端大模型强依赖网络。是否本地化部署直接决定技术方案和成本，必须在早期问清。<strong>你能用</strong>：任何 B 端/工业 AI 项目，开工前先问一句"现场有没有网、能不能连公网"——答案是"不能"时，整个方案要换成本地化/局域网部署。

### 5. "这东西到底有什么用"——供应商的诚实困惑也是一种价值
供应商中途坦白："我一直不明白这个东西有什么用""如果只是做 web 端，这件事我们早就做完了""你又不是要做投屏，小智在你这个场景上没什么用。"他甚至建议客户：电视装个浏览器、电脑接个大屏做 web 端就行，不用搞硬件。
<strong>为什么重要</strong>：好的售前会质疑需求本身，而不是客户要什么就卖什么——他在帮客户省钱、也在逼出真需求（最后逼出"要语音灵活驱动、不是被动展示网页"）。<strong>你能用</strong>：当客户的需求用更简单的方案就能满足时，敢于说"你其实不需要买这个"——这反而建立信任，也常常是逼出真需求的最后一脚。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：把"方案语言"翻译回"问题语言"</strong>
- 核心意思：客户说"投屏/连电视"是方案，背后的问题是"语音调出报表显示在大屏"。
- 你可以怎么用：客户给名词/方案时，回一句"你最终是想达到什么效果"，把方案拆回目标。
- 不要误读成：客户说什么就照做——顺着错方案做会交付错东西。

<strong>方法 2：用"收敛型追问"把需求逼成规格</strong>
- 核心意思：靠"最终传给硬件的是什么""有几种类型——文字、图表、还有吗"这种问题，把模糊需求收敛到可报价。
- 你可以怎么用：准备一组把形容词逼成名词/枚举的问句，对话里反复用。
- 不要误读成：一次问完——本场就是问了七八轮才逼清，需要耐心和反复确认。

<strong>方法 3：先对齐"固定脚本还是动态可配"</strong>
- 核心意思：脚本便宜但僵、大模型驱动灵活但贵，这是报价和工程量的分水岭。
- 你可以怎么用：需求澄清阶段直接问客户要哪种，写进方案。
- 不要误读成：默认客户要最灵活的——也别默认最便宜的，必须明确问。

## 📅 7 天作业

- <strong>Day 1</strong>：找一个你手上最模糊的需求，写下客户原话，再把它翻译成"输入是什么 / 输出是什么 / 有几种情况"。
- <strong>Day 2</strong>：为这个需求准备 5 句"收敛型追问"，专门把客户的形容词逼成名词和动作。
- <strong>Day 3</strong>：搭一个最小 demo 设想：用一个语音设备（或手机语音）作 client、你电脑某个能力包成 MCP server，让语音能打开它。
- <strong>Day 4</strong>：把这个需求拆成"固定脚本版"和"大模型动态驱动版"两套方案，各估一下工程量和报价。
- <strong>Day 5</strong>：列出这个项目所有"网络/部署"约束——要不要断网、要不要本地化、是不是内网，写成一页前置问题清单。
- <strong>Day 6</strong>：找一个客户的"我要 X"需求，练习一次"你其实不需要 X，更简单的办法是 Y"的对话。
- <strong>Day 7</strong>：把这次的澄清过程复盘成一张"需求 → 真需求 → 技术架构"的图，沉淀成你自己的售前模板。
