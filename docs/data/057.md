## ⏱ 如果只读 10 分钟（7 条带走判断）

1. <strong>这类 AI 玩偶的"硬件"早就不值钱，值钱的是表情美术 + 软件交互 + 长期记忆。</strong> 方案商明说外观结构一般不包、表情让"高价设计师"做，自己只做"纯硬件方案 + 代码 + 服务端 + App"。硬件是壳，差异在软件和美术。

5. <strong>本地交互 vs 联网交互要分开设计——这决定了功耗和体验。</strong> 光线明暗触发"害怕"表情、速度/摇晃触发反应=本地不联网；问问题/对话=联网调云端 API。把"不联网也能玩"的部分做厚，体验和续航双赢。
6. <strong>响应速度被方案商当成核心卖点，但其口径要看清。</strong> 自称调 API 能做到 1 秒左右、"市面最快"；并主动拆穿"小智 500ms 是本地部署堆卡、和调 API 不是一回事"。这是难得的诚实，也是甲方该追问的技术口径。
7. <strong>回扣、付款分期是这类项目的"潜规则水面"。</strong> 甲方主动问负责人奖金、提"谈成私下给回扣"；付款常见 1:5:5、5:3:2 分期。做 B2B 硬件外包，这些是绕不开的桌下变量，听到要心里有数。

## 📚 深挖：6 个值得单独学的知识点

### 1. 一台 AI 玩偶被拆成六块：谁包、谁不包、钱花在哪
方案商把产品拆得很清楚：<strong>外观结构</strong>（一般不包，客户自带设计）、<strong>表情美术</strong>（不做，让"高价设计师"客户定制，因为表情吸引小朋友最关键）、<strong>固件/硬件/结构/服务端</strong>（4 个人力、自己做）、<strong>代码+服务端+通信接口+App</strong>（包，是核心交付物）。报价约 6–7 万（除美工）。
<strong>为什么重要</strong>：这张"分工图"揭示了 AI 硬件的真实价值分布——硬件集成是可外包的标准件，真正稀缺的是美术 IP（"找高价设计师"）和软件交互。谁掌握不可外包的那块，谁就掌握定价权。
<strong>你能用</strong>：做这类产品前，先画出"六块分工表"，标出每块自己做还是外包、各占成本多少，看清自己到底掌握了哪块护城河。

### 2. 续航是第一性约束，倒着锁死电池/体积/成本
全场最硬的技术讨论都绕着续航：方案商先问"你要待机续航还是使用续航"——这是行家才会区分的问题。<strong>待机低功耗</strong>约 1000–1500mAh 够用；想撑"白天 11–12 小时全程能聊"要 <strong>2000–3000mAh</strong>，但挂件体积根本塞不下大电池。所以最终落到"日常使用 2–3 小时交互、1000–1500mAh"的折中。
<strong>为什么重要</strong>：这是一条单向锁死的约束链——使用场景→交互时长→电池容量→机身体积→成本。挂件这种"要小、要轻、要挂书包上"的形态，天然和"大电池长续航"矛盾。不先把这条链想清楚，后面所有参数都是空中楼阁。
<strong>你能用</strong>：设计任何便携 AI 硬件，第一个动作不是选芯片，是写清楚"用户一天用几小时、什么场景、能接受多大多重"，用它倒推电池，再倒推体积和成本。

### 3. 本地交互 vs 联网交互——一条分界线决定功耗和体验
方案商和甲方一起厘清了一条关键边界：<strong>本地不联网</strong>就能做的——光线明暗触发"害怕"表情、速度/摇晃/奔跑骑车触发反应、简单语音控制（"就是没有 AI 时的那种语音识别"）；<strong>必须联网</strong>的——问问题、对话、理解语义（调云端大模型 API）。待机时基本不需要联网。
<strong>为什么重要</strong>：这条分界线同时决定了三件事——功耗（本地省电）、体验（本地零延迟）、成本（联网要服务端+token）。把"不联网也好玩"的部分做厚，是续航和体验双赢的关键设计选择。
<strong>你能用</strong>：列出你产品的交互清单，逐条标"本地/联网"，尽量把高频、即时反馈的交互放本地，把"需要智能"的留给云端。

### 4. "100 种交互指令"——怎么拆穿竞品宣传话术
甲方拿对标品（疑似 Looi/Ropet，ASR 作"艾莱克欧/Alex"）的"100 种交互指令"来问，方案商的判断很冷静："它现在还没走向市场、还在众筹生产阶段，应该没有 100 种，可能是前期宣传"。并补充：自定义指令工作量大、按数量收费；不自定义就按支持的指令数量算。
<strong>为什么重要</strong>：这演示了一种通用的"参数祛魅"能力——看竞品宣传先分清<strong>已上市实测</strong>还是<strong>众筹/PPT 阶段的营销数字</strong>。众筹期的参数是用来融资和造势的，不等于交付能力。
<strong>你能用</strong>：调研竞品时，给每个亮眼参数标注来源状态（已量产/众筹中/纯宣传），只对"已上市实测"的参数当真。

### 5. 响应速度的"口径战"：1 秒、500ms 到底怎么比
方案商主动拆穿了一个常见误导："小智能做 500ms 是因为它本地部署了很多卡；我们说的是调用 API 的形式，1 秒左右、市面最快。"——明确区分了<strong>本地堆算力</strong>和<strong>调云端 API</strong>两种实现路径的速度不可直接对比。
<strong>为什么重要</strong>：这是难得的技术诚实，也教会甲方一把尺子：比"响应速度"必须先问"在什么架构下测的"。本地部署堆卡的 500ms 和调 API 的 1 秒，成本结构、可量产性完全不同，拿来直接 PK 是田忌赛马。
<strong>你能用</strong>：评估任何"我们最快"的硬件性能宣称，先追问"在什么部署架构、什么成本下达到的"，否则数字没有可比性。

### 6. 让甲方先交"功课"才能精准报价——方案商的反控价术
方案商反复强调：要精准报价，甲方得先给三样东西——<strong>结构设计/外壳建模</strong>（看"空箱能塞多少东西"，还要考虑手部传动机构占空间）、<strong>小程序需求文档</strong>（要详细，决定服务端/App 工作量）、<strong>长期记忆/社交关系等功能清单</strong>。在这些没到位前只能给"6–7 万"的粗略区间。
<strong>为什么重要</strong>：这揭示了硬件外包报价的真实逻辑——报价精度取决于需求颗粒度。方案商用"先给功课再精准报价"既保护自己（避免低报漏项），也筛选甲方诚意。同时点出一个易被忽略的硬件细节：<strong>会动的手需要传动机构，要占内部空间</strong>，影响电池和板子布局。
<strong>你能用</strong>：找硬件外包时，主动先把结构、功能清单、交互文档备齐再谈，既能拿到精准报价，也显示你是靠谱甲方；同时记得把"机械传动占空间"这类隐性约束提前摊开。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：用"使用场景"倒推硬件参数</strong>
- 核心意思：先定"用户一天用几小时、什么场景、能接受多大多重"，再倒推电池→体积→成本。
- 你可以怎么用：谈续航前先写一句话使用画像（如"挂书包、日常交互 2–3 小时、待机一整天"），让对方据此报方案。
- 不要误读成：一上来就追求"待机一整天还全程能聊"——那是 2000–3000mAh 电池，挂件根本塞不下。

<strong>方法 2：交互清单分"本地/联网"两栏</strong>
- 核心意思：高频即时反馈走本地（省电零延迟），需要语义智能的走云端 API。
- 你可以怎么用：把所有交互列成两栏表，尽量加厚本地那栏。
- 不要误读成：什么都上云才显得"AI"——纯展示/情绪反馈类交互上云只是徒增功耗和成本。

<strong>方法 3：竞品参数按"来源状态"打折看</strong>
- 核心意思：区分已量产实测、众筹中、纯宣传三种参数可信度。
- 你可以怎么用：给每个对标参数标来源，只对已上市实测的当真。
- 不要误读成：竞品官网写了就是能做到——众筹期数字常是营销话术。

## 📅 7 天作业

- <strong>Day 1</strong>：把你想做的 AI 硬件拆成"六块分工表"（外观/美术/固件硬件/结构/服务端/App），标出每块自己做还是外包。
- <strong>Day 2</strong>：写一句话"用户使用画像"（场景+每天交互时长+可接受体积重量），据此估算需要多大电池。
- <strong>Day 3</strong>：列出产品全部交互，逐条标"本地/联网"，统计哪栏更厚。
- <strong>Day 4</strong>：找 2–3 个对标品（如 Looi/Ropet），给它们每个亮眼参数标注来源状态（已量产/众筹/宣传）。
- <strong>Day 5</strong>：起草一份"小程序/App 需求文档"骨架（长期记忆、社交关系、用户数据管理至少各一段），这是外包报价的关键输入。

- <strong>Day 7</strong>：设计付款分期方案（如 5:3:2），把每期对应的交付里程碑写死，作为履约抓手。
