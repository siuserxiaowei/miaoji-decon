> 场景：一个 AI 陪伴/玩具硬件项目（带小程序 + 管理端 + 声纹/闹钟/声音克隆的硬件盒子）的甲乙方过单会。甲方（产品负责人/老板那条线）拿着自己测出来的约 13 个问题，逐条对着乙方（开发，"胡工/普工"等）评审：哪些是 bug 必改、哪些是合同写过必须做、哪些是"新功能"要重新谈。会议本质是一场<strong>围绕"合同边界"的验收博弈</strong>，不是技术分享会。
>
>
> ⚠️ 如果飞书原 AI 摘要把这场总结成"客户改 bug 清单"——只接住了一半。真正值得学的是：<strong>一个改了很多版、已经亏本的外包项目，乙方是怎么用"合同写没写"这把尺子，逐条挡掉范围蔓延（scope creep）的</strong>，以及甲方是怎么一步步退让到"能接受就接受"的。

## ⏱ 如果只读 10 分钟（7 条带走判断）

1. <strong>"合同写没写"是乙方唯一的护身符</strong>。全场乙方反复用同一句话挡需求："如果合同没写那我也不搞"。改了很多版、项目已亏本的情况下，这句话是把范围蔓延（scope creep）摁住的唯一武器——而且它真的有效，第 10 条统一管理就是靠"合同没写"被甲方自己撤掉的。
2. <strong>甲方的需求会自然膨胀，且常包装成"小改动"</strong>。"统一管理所有用户智能体"听起来像顺手，实则要新写一个 web 端、要拉通所有用户数据——是典型的"一句话需求，一个月工时"。乙方的应对是：先把它拆开问清楚到底要什么，再判定是不是新功能。
3. <strong>接支付 = 工作量黑洞，能不接就别接</strong>。声音克隆若直接给 C 端用户自助购买，要打通火山引擎 + 微信两个平台的支付、对账、权责，乙方直接说"我们可能要亏死"。最终方案绕开支付：<strong>公司先批量买一批音色 ID 存进表，用户领一个、从表里取一个</strong>——用"预购库存池"换掉"实时支付"。
4. <strong>样品到量产，最少半年，不是"给工厂就能产"</strong>。甲方老板想年前（农历春节前）上市，乙方直接泼冷水：开模 30–60 天只是开始，还要小批量、各种测试、驻场培训工厂——半年都说少了。这是硬件创业最常见的认知差。
5. <strong>AI 玩具这行"宣传占 70%"</strong>。乙方原话：产品大同小异、负载（友商/对标产品）也没什么功能，真正决定卖不卖得动的是宣传和 IP。有现成粉丝群体的玩家（如做智能家居转做硬件的），产品差一点也照样卖得动。
6. <strong>第一次响应慢，是"现场实时生成 + 未缓存"的通病</strong>。闹钟首响时铃声和文案间隔久，因为第一次才去请求服务端生成文案、之后才缓存。这是所有"端 + 云实时生成"产品的共性延迟，优化方向是预请求/预缓存，但要小心两段语音重叠播放。
7. <strong>验收要"改一个验一个"，别攒一堆</strong>。甲方主动提出节奏：乙方改完一条就通知、甲方验一条，"不然一堆弄起来到时候比较麻烦"。这是把交付风险拆小、避免最后总验收翻车的朴素工程管理。

## 🎯 这篇真正能学什么

这不是一份 bug 列表，是一堂<strong>外包/甲乙方协作的实战课</strong>。值得学三件事：①<strong>范围管理</strong>——当一个项目已经亏本、改了很多版，乙方如何用合同把"该做的"和"新功能"切开，并且不撕破脸地让甲方自己撤需求；②<strong>技术决策的成本权衡</strong>——为什么"接支付""统一管理""自动买音色"这些听起来顺手的需求，工程上是黑洞，以及绕开它们的替代设计（库存池替代实时支付）；③<strong>硬件产品的现实约束</strong>——量产周期、AI 玩具行业"宣传/IP 大于功能"的真相。读完别记那 13 条 bug，记"如何谈判范围"和"哪些需求是成本陷阱"这两张图。

## 📚 深挖：8 个值得单独学的知识点

### 1. "合同写没写"——外包项目的唯一止血点（这是全场的"道"）
这个项目"改了很多版""完全亏本了""现在暂时不想改"。在这种情况下，乙方挡掉范围蔓延的全部武器就一句话：<strong>"如果合同没写那我也不搞。"</strong> 第 1 条登录注册、第 9 条 TTS 三参数、第 13 条声音克隆，乙方都先去翻合同/翻群里的聊天记录确认"写过没"。最经典的是第 10 条"统一管理"：甲方想要、乙方判断改动大，最后是乙方引导甲方一起查合同，确认"确实没说需要统一放管理端"，于是<strong>甲方自己说"那就先不要"</strong>。
<strong>为什么重要</strong>：亏本项目不能再无限投入，但直接拒绝会撕破关系。用"合同"做客观第三方裁判，既守住边界又不伤面子。
<strong>你能用</strong>：任何协作/外包，把"是否已约定"作为每个需求的第一道闸；模糊的当场拉聊天记录/合同对齐，别等结账时算账。

### 2. 范围蔓延（scope creep）长什么样：一句话需求 = 一个月工时
第 10 条"能不能把所有用户的智能体统一展示在管理端，方便我们排查问题"——甲方觉得是顺手的小功能。乙方拆开后点破：这要"写一个 web 端，显示所有用户的智能体"，要拉通全部用户数据权限。甲方真实诉求其实很小："用户有问题时我能看到他那边什么情况，而不是盲盒。"
<strong>为什么重要</strong>：甲方常把"大改动"用"小需求"的语气说出来；乙方的价值在于<strong>把需求翻译成工时</strong>，再让甲方在成本面前重新权衡。
<strong>你能用</strong>：听到"能不能顺便……"先别答能不能，先问"你到底想解决什么问题"，往往有更便宜的实现。

### 3. 接支付是成本黑洞——用"库存池"绕开实时支付
第 13 条声音克隆，若按"用户自助买音色"做，要打通<strong>火山引擎付款 + 微信小程序付款两个平台</strong>，加对账、用户权责，乙方判断"3 个 API 起步""我们可能要亏死"，还要拿公司资质去申请、审批周期长。最终落地方案：
- 公司<strong>先批量买一批音色</strong>（买 10/20 个，含空白音色）存进一张 speaker ID 表；
- 用户领取时<strong>从表里取一个</strong>填入、自己录音使用；
- 库存低于阈值（如少于 50 个空白音色）<strong>告警人工补货</strong>，而不是让用户实时去火山自动买。
<strong>为什么重要</strong>：这是"用库存预购替代实时交易"的经典工程取舍——把一个跨平台、跨权责的复杂实时系统，降维成一张表 + 人工补货。
<strong>你能用</strong>：凡是"要不要接第三方支付/自动采购"的需求，先问能不能用"预付库存池 + 告警补货"绕过去，往往省掉 80% 工作量。

### 4. 样品 ≠ 量产：硬件时间表的认知差
甲方老板想"年前（农历春节前）量产上市"。乙方逐条拆：开模 30–60 天只是起点，之后要小批量、各种测试、<strong>驻场培训工厂教他们怎么做</strong>，还可能要再改几版，"最少半年，不是说你这个样品给到工厂人家直接复刻出来就能上市"。
<strong>为什么重要</strong>：软件人/产品人常低估硬件量产链条；"样品能用"和"能稳定量产"之间隔着开模、试产、测试、产线培训。
<strong>你能用</strong>：做任何硬件项目，对外承诺时间表按"样品完成 + 半年起"沟通；把"驻场培训工厂"这种隐藏工序提前列入计划。

### 5. AI 玩具行业的真相："宣传占 70%，IP 很重要"
乙方的行业判断：市面产品"大同小异"，对标的负载"也没什么功能，就是靠宣传"。真正卖得动的逻辑是 <strong>IP + 已有粉丝池</strong>——举例"虾线科技"（ASR 识别，名字待核）做车载机器人，靠延续 IP；还有做智能家居的客户转做硬件，"产品比别人差一点没关系，宣传到位 + 用户粘度在那"照样卖。深圳/汕头是产业集群，科创学院有院士带队，政府还有 AI 玩具产业基金/计划。
<strong>为什么重要</strong>：在一个"功能同质化"的卷行业，胜负手不在多一个功能，在分发（宣传）和信任资产（IP/粉丝）。
<strong>你能用</strong>：做 AI 硬件别堆功能清单，先想清楚 IP/宣传/已有用户池从哪来；没有粉丝池就别赌纯功能。

### 6. "端+云实时生成"的首响延迟，本质是缓存策略
第 4 条：闹钟第一次响时，铃声和文案提示之间间隔久。原因是<strong>第一次才去请求服务端生成文案</strong>，之后才缓存在硬件里，所以后续就快。优化讨论：能不能把请求从"铃声结束后"提前到"铃声开始响时"就发出去？风险是——服务端语音下发可能和本地铃声<strong>重叠播放</strong>（两个声音混在一起）。结论是先"把铃声时长缩短一点"试。
<strong>为什么重要</strong>：所有"实时云端生成 + 本地播放"的产品都有这个首响延迟；预请求能提速但要处理时序冲突（两段音频不能同时响）。
<strong>你能用</strong>：做实时生成类功能，先想"能不能预请求/预缓存"，但务必先排好播放时序，别让预加载导致声音打架。

### 7. C 端只暴露用户看得懂的，高级配置出厂锁死
第 9/10 条：管理端有人设、模型配置、ASR/TTS、识别速度、角色语速、音调等一堆参数。甲方明确："小程序端不需要他们改这些，估计他们也看不懂，太多了。我们出厂默认设置好，用户只选角色。"用户只保留<strong>昵称、音色、识别速度</strong>这几个基础项；其余红色高级项（模型、人设）在管理端由管理员设默认值，"统一掉，不让他们改来改去改坏了，维护起来麻烦"。
<strong>为什么重要</strong>：这是产品设计的"暴露面控制"——给用户的旋钮越少，支持成本和出错率越低。
<strong>你能用</strong>：盘点你产品暴露给终端用户的配置项，凡是"用户看不懂、改了会坏"的，一律出厂默认 + 隐藏，只留几个友好选项。

### 8. 验收节奏："改一个验一个"而不是攒批总验
甲方主动定的协作节奏："你们改好之后及时跟我们说，我们验，一个一个改完一个验一个，不然一堆弄起来到时候比较麻烦。"乙方也把工作排序：先修 bug（简单的几天搞定），新功能/要申请的东西（如声音克隆、支付）单独做计划评估工时。
<strong>为什么重要</strong>：把交付拆成小批次、即时验收，能在早期暴露理解偏差，避免最后总验收时大面积返工。
<strong>你能用</strong>：任何多条改单的协作，约定"逐条交付逐条验收"，并把"bug 类"和"新功能类"分两个时间盒（time-box）管理。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：每条需求先打"合同标签"再动手</strong>
- 核心意思：把每个需求分成 bug必改 / 合同内必做 / 新功能再谈三类，用合同/聊天记录当裁判。
- 你可以怎么用：收到需求清单先逐条标签化，模糊的当场对齐，新增的单独报价/报工时。
- 不要误读成：用合同当挡箭牌一概拒绝——bug 和合同内的该改还得改，挡的只是"没约定的新功能"。

<strong>方法 2：用"库存池 + 告警补货"替代"实时第三方支付/采购"</strong>
- 核心意思：把跨平台实时交易降维成"公司预购一批资源存表，用户领取时取用，低于阈值告警人工补货"。
- 你可以怎么用：凡涉及第三方付费资源（音色、API 额度、号码等），优先评估预购库存池方案。
- 不要误读成：永远不接支付——库存池适合量不大/可控的场景；真要做用户自助大规模购买，还是得正经接支付。

<strong>方法 3：把"顺手小需求"翻译成工时再让对方决策</strong>
- 核心意思：甲方常把大改动说成小需求，乙方先拆出真实工时和真实诉求，再让对方在成本面前权衡。
- 你可以怎么用：听到"能不能顺便"，先回"这背后要做 X/Y/Z，大概多少工时，你真正想解决什么"。
- 不要误读成：用"工作量大"吓退所有需求——目的是帮对方找到更便宜的实现（如统一管理→其实只要能查单个用户状态）。

## 📅 7 天作业

- <strong>Day 1</strong>：把你手上一个协作/项目的所有待办需求逐条打标签——已约定 / 新增 / 模糊；模糊的列出来今天就去对齐。
- <strong>Day 2</strong>：挑一个"听起来顺手"的需求，把它真实拆成工时清单，看是不是被低估的范围蔓延。
- <strong>Day 3</strong>：找你产品里一个"接第三方支付/采购"的需求，设计一个"库存池 + 告警补货"的替代方案并估算省下的工作量。
- <strong>Day 4</strong>：盘点你产品暴露给终端用户的所有配置项，圈出"用户看不懂/改了会坏"的，规划成出厂默认 + 隐藏。
- <strong>Day 5</strong>：如果你做硬件，画一条从样品到量产的真实时间线（开模 / 试产 / 测试 / 产线培训），看你对外承诺的时间表现实吗。
- <strong>Day 6</strong>：找一个"实时云端生成"的功能，评估能否预请求/预缓存提速，并列出可能的时序冲突（如音频重叠）怎么处理。
- <strong>Day 7</strong>：和一个长期合作方约定"逐条交付逐条验收"的节奏，并把 bug 类和新功能类分成两个时间盒分别排期。
