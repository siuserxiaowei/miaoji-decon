## ⏱ 如果只读 10 分钟（5 条带走判断）

1. <strong>甲方以为"纯软件"的需求，其实卡在硬件资源上</strong>。徐总想让设备"什么都能识别"（检查作业 + 路上随便看到一棵树就能问），乙方明确回："场景太大，硬件资源匹配不了、数据库存储也撑不住，只能固定几个场景。" 这是 AI 硬件项目最典型的认知错位。
2. <strong>多场景识别 = 多套提示词，不是一份提示词通吃</strong>。乙方解释：提示词要"给角色、给场景、框定方向"；要从"检查作业"切到"识别物体"，得<strong>重写一套提示词</strong>。换提示词理论上能支持多任务，但需要先把场景数量定下来。
3. <strong>设备的"表情/眼睛动画"在吃硬件资源</strong>。后段卡顿、眼球转动有彩影，被判断为"硬件资源不够 / 真品（帧率？）做太高"。解决思路：只让黑底上的"白色眼睛"局部变化，不要整屏图都动——降资源占用。
4. <strong>打板供应链没有"加一天就出片"这回事</strong>。徐总踩坑：供应商口头说 PCB 2-3 个工作日没问题、贴片加一天，实际 SMT 要排期，下单数量小就更不优先。结论：小批量样品的交期靠"关系 + 加急 + 排期"，不能信一口价时间承诺。
5. <strong>贴片只是中间一环，后面还有 CNC 外壳 + 组装 + 二次集成</strong>。徐总的 AI 促销员要洗铝合金外壳（CNC）→ 装板进壳 → 发去江西装到饮水机外面。"不是贴出来就完事儿了"——做硬件要算的是整条链路的交期，不是单点。

## 📚 深挖：4 个值得单独学的知识点

### 1. 甲方的"软件需求"被硬件资源反向卡死——AI 硬件最常见的认知错位
徐总的设想很自然："硬件只负责拍照、传给大模型、把返回文本转语音读出来，剩下都跟硬件没关系。" 但乙方反复纠正：要做到"小孩看到什么都能问"（开放域识别），<strong>数据库存储和硬件资源根本撑不住</strong>，"只能固定几个场景"。
- <strong>为什么重要</strong>：这是软硬件协作里最经典的摩擦点。甲方按"云端大模型无所不能"的心智提需求，乙方面对的是端侧的算力/存储/带宽现实。开放域 vs 限定域，是能不能落地的分水岭。
- <strong>你能用</strong>：接需求时，先问"要支持几个场景"，把开放式的"什么都能识别"压成可枚举的几个，再谈资源和报价。

### 2. 提示词 = 角色 + 场景 + 方向，多任务要多套提示词
乙方对"解析方向固定"的解释很干净："最好的方式是给他一个角色、给他一个场景，把它框定在一个方向。" 现在框的是"检查作业"；若要切到"辨别这是什么物体"，得<strong>重新写一个提示词</strong>。换提示词理论上能让同一设备支持两种任务，但默认交付的那一份只服务"检查作业"。
- <strong>为什么重要</strong>：这把"提示词工程"讲成了产品交付语言——能力边界是被提示词框出来的，不是模型自动全能。客户常误以为"一份提示词应该啥都会"。
- <strong>你能用</strong>：交付 AI 功能时，明确告知客户"每个场景对应一套提示词"，把场景清单当成需求确认的核心交付物。

### 3. 端侧表情动画的资源优化：只动局部，不动整屏
卡顿和"眼球转动有彩影"被归因为硬件资源不够 / 帧率（"真品"疑为 ASR 误写）做太高。优化思路很具体：<strong>黑色底永远不变，只让两个白色眼睛发生变化</strong>；如果整幅屏幕的图都去动，资源占用就大。另外提到待机时"眯缝眼"是正常状态，听问题/回答之间的过渡表情在资源紧张时可以砍掉。
- <strong>为什么重要</strong>：这是端侧 UI 动效的通用省资源原则——区分"必须动的小区域"和"可以静止的大背景"，对低算力硬件尤其关键。
- <strong>你能用</strong>：做嵌入式/低端硬件的动画时，先拆"哪块非动不可"，把背景固化、只重绘局部。

### 4. 硬件交付是一条链，不是"贴出片"就结束
徐总把自己的真实链路摊开了：打板 → SMT 贴片（要排期，不是加一天就出）→ CNC 铝合金外壳（另找加工厂在洗）→ 装板进壳组装 → 发江西装到饮水机外壳上。他踩的坑：在供应商网站下单"折腾了三天"，被告知 PCB 2-3 天没问题、贴片加一天就行，实际"说不定"、要排期；下单量小就更不被优先；量上来了价格才下得去、才好加急。
- <strong>为什么重要</strong>：硬件项目的交期风险不在单点，而在链路衔接和供应商口头承诺的水分。把单点乐观估计串起来，整条链的实际交期会被排期和返工拉长数倍。
- <strong>你能用</strong>：排硬件计划时，按"打板→贴片→外壳→组装→集成"逐段列交期和负责人，对每个口头时间承诺打折，并预留排期缓冲。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：把"什么都能识别"翻译成"固定几个场景"再承接</strong>
- 核心意思：开放域需求在端侧硬件上落不了地，先把场景枚举出来。
- 你可以怎么用：需求评审时直接问"具体要支持哪几个场景"，逼客户收敛。
- 不要误读成：一口回绝客户"做不到"——是把需求<strong>收敛</strong>到硬件能扛的范围，而不是砍掉。

<strong>方法 2：按场景拆提示词，把提示词清单当交付物</strong>
- 核心意思：一套提示词框一个场景，多场景就多套。
- 你可以怎么用：把"要几套提示词"写进需求确认单，避免事后扯皮。
- 不要误读成：以为换提示词是零成本——每套都要单独写、单独占资源。

<strong>方法 3：供应商时间承诺一律打折 + 自己留排期缓冲</strong>
- 核心意思："加一天就出片"基本不可信，SMT 要排期、小单更靠后。
- 你可以怎么用：把每段交期 ×1.5~2，关键节点提前问"是否要排期、能否加急"。
- 不要误读成：换一家更快的供应商就解决了——徐总自己也说"匆忙找另一家不一定有现在这家做得好"，质量优先于赶时间。

## 📅 7 天作业

- <strong>Day 1</strong>：把当前这个 AI 硬件要支持的场景列成一张清单（检查作业 / 物体识别 / …），逐个标"是限定域还是开放域"。
- <strong>Day 2</strong>：为"检查作业"和"物体识别"各起草一套提示词（角色+场景+方向），验证同一设备换提示词能否切任务。
- <strong>Day 3</strong>：实测表情动画——对比"整屏动"和"只动白眼睛"两种方案的资源占用和卡顿。
- <strong>Day 4</strong>：把拍照→传输→大模型→文本→语音这条链路画成时序图，标出哪一步"不响应语音"是正常的，并设计过渡安抚话术（"让我看看"）。
- <strong>Day 5</strong>：把打板→贴片→CNC外壳→组装→集成整条供应链列出来，每段写负责人和真实交期。
- <strong>Day 6</strong>：找当前打板/贴片供应商书面确认 SMT 排期规则和加急条件，把口头承诺落成文字。
- <strong>Day 7</strong>：评估"下单数量"对贴片价格和交期的影响，算出一个能换到加急资格的批量门槛。
