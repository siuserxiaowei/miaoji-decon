## 📚 深挖：6 个值得单独学的知识点

### 1. "不同角色 = 不同模型"是错的：一个通用大模型 + 角色设定就够
甲方反复纠结："孩子角色、情人角色、医生角色，是不是每个角色对应一个独立的模型？"乙方明确回答："这个没有必要……通用的大模型就足够了，角色你可以给它定义。"
<strong>为什么重要</strong>：这是 AI 应用最普遍的认知误区。大模型是"通用能力底座"，"角色"是在它之上用系统提示词/人设/知识库塑造出来的表现层，不是另起一个模型。把"角色"误当"模型"会凭空放大开发量和成本。
<strong>你能用</strong>：做多角色对话产品，先用一个通用大模型 + 不同的角色提示词跑通，验证体验后再考虑是否需要针对垂类（如医疗）做微调/知识库。

### 2. 通用大模型怎么选：速度 vs 深度思考的取舍
甲方问"通用大模型一般用哪个"，乙方给的候选是<strong>通义千问（Qwen）、DeepSeek、豆包</strong>，都可以。但点出一个关键体验差异：DeepSeek <strong>开了"深度思考"就慢</strong>（它要先"想"），关掉深度思考就快，但回答效果会打折。
<strong>为什么重要</strong>：对"情感陪伴/实时对话"这种强调响应速度的场景，模型选型的核心矛盾不是"谁更聪明"，而是"延迟 vs 质量"。深度思考型模型不适合要求秒回的陪伴对话。
<strong>你能用</strong>：做实时对话产品，优先选不强制深度思考、低延迟的模型档位；需要复杂推理的场景再单独走"思考"模式。（注："QV 模型""充 1000 万"系 ASR 识别，结合上下文应指通义千问 Qwen，金额表述存疑，需核实。）

### 3. "像小智那样快" = 本地部署 + 多张显卡，这是真实成本台阶
甲方羡慕"小智"（小智 AI 智能体/音箱）的响应速度，问怎么做到。乙方点破：要那么快就得<strong>本地部署</strong>——把大模型跑在本地，靠 GPU（显卡）提供算力。还给了个口径："小智官方那个用了八张 3090，后面可能都不止了。"
<strong>为什么重要</strong>：很多人以为"快"是软件优化问题，其实在大模型场景里，"快"很大程度是<strong>算力堆出来的</strong>（本地部署省掉网络往返 + 充足显卡并行）。这是一条实打实的硬成本：八张 3090 是一笔不小的硬件投入。
<strong>你能用</strong>：客户提"要像 XX 一样快"时，先把"快"翻译成"本地部署 + N 张显卡 + 运维"的成本清单给他看，让他知道速度是花钱买的。（"八张 3090"为现场口径，需核实。）

### 4. 私有化 + 多地部署：成本怎么从"调 API"滚到"自建机房"
甲方的需求一步步加码：数据要存自己服务器 → 要本地部署保证速度 → 每台机器要<strong>固定公网 IP</strong> 对外提供服务 → 甚至"每个省一台机器"。乙方的回应是"看用户量"——量不大不用每省都铺，一张显卡也行。
<strong>为什么重要</strong>：这是一条典型的<strong>成本指数级上升路径</strong>。"调一个云端 API"几乎零基建；一旦要"私有化 + 低延迟 + 多地服务"，就变成自建算力基础设施：每个节点都要显卡、公网 IP、运维、扩容。需求每加一条，成本上一个台阶。
<strong>你能用</strong>：给定制硬件/私有化项目报价时，把"私有化""低延迟""多地服务"这三个词单独拎出来定价——它们才是成本大头，而不是"做个机器人"本身。

### 5. 甲方的合作三段漏斗：验实力 → 验诚信 → 签约打钱
甲方把自己的决策逻辑讲得很白："第一件事就是判断你和我之间的实力，你判断我的实力，我判断你的能力""第二个判断你的诚信度""我喜欢把事情摊开讲，行我们一起干，不行你告诉我，我去找别人"。落到具体动作，就是一句反复出现的"<strong>给个样品/复刻一个给我看效果</strong>"。
<strong>为什么重要</strong>：这是 B 端定制生意里非常标准的信任建立路径——<strong>没有信任就没有预付款</strong>。甲方甚至说"你没有固定的地方，对我来讲信誉度不高"，说明物理可信度（有没有固定场地）也是他的判断维度。
<strong>你能用</strong>：面对要打大额预付款的甲方，主动提供"低成本可验证的样品"作为信任敲门砖；同时补强自己的"可信度证据"（固定场地、过往案例、可远程演示的后台）。

### 6. 合作形式要先讲清：员工制 vs 承包制，以及版权安全
甲方反复追问对方"<strong>你是以我员工的形式承载，还是以承包的形式承载</strong>"，并立刻接出关键顾虑："如果你承包，那创意代码以及这些东西属于我们公司，怎么安全地保障？"他还提供办公场地、设备费、场地费，"我来支撑你们团队的开发"。
<strong>为什么重要</strong>：合作形式（雇佣 vs 承包）直接决定<strong>成果归属、版权、保密责任、付款结构</strong>。甲方先谈这个再谈技术，说明他清楚"权责归属"是定制开发里最容易扯皮、最该前置谈死的事。
<strong>你能用</strong>：接定制开发单，第一轮就把"合作形式 + 代码/IP 归属 + 保密 + 分阶段付款"写进框架，别等做完了才补。这既保护你，也让甲方觉得你专业、靠谱。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：把"角色"和"模型"分开——先通用大模型 + 人设跑通</strong>
- 核心意思：多角色对话不需要多个模型，一个通用大模型 + 角色提示词/知识库即可。
- 你可以怎么用：MVP 阶段先用 Qwen/豆包 + 角色 prompt 验证体验，再决定要不要微调。
- 不要误读成：通用模型永远够用——特别细分的垂类（如医疗）仍可能需要知识库/微调，只是不是"换个模型"那么简单。

<strong>方法 2：把客户的"要快"翻译成可报价的成本清单</strong>
- 核心意思："像小智一样快"= 本地部署 + 多张显卡 + 公网 IP + 运维，是花钱买的。
- 你可以怎么用：客户提速度要求时，直接给一张"基建成本台阶表"，让他自己权衡要不要为速度付钱。
- 不要误读成：所有项目都得本地部署——用户量不大时，云端/单卡就够，别过度工程化。

<strong>方法 3：用"低成本样品"破信任僵局</strong>
- 核心意思：观望型甲方靠"看得见的效果"建立信任，样品/复刻/远程演示比反复解释有效。
- 你可以怎么用：准备一个可快速复刻、可远程连后台演示的 demo，作为标准破冰物料。
- 不要误读成：免费无限做样品——可以收成本费（甲方自己说"多少钱我都可以支付""今天就给你打钱"），样品是收费的信任投资。
