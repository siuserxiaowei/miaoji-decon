## ⏱ 如果只读 10 分钟（7 条带走判断）

1. <strong>"用小智那套"和"自建服务端"是两条命，越早分清越省钱</strong>。乙方反复强调：小智官方那套适合个人/几台玩、快速低价上线；只要你想商用、上量（哪怕一百台）、要自定义能力，就必须自己有云端去调外部大模型。这是整通话的主轴。
2. <strong>情绪感知 / 音色识别这类能力，小智那套（VAD→ASR→大模型→TTS）天生做不了</strong>——因为语音先转成文字，声学特征（语调、语速、情绪）在转文字那一步就被丢掉了。要做就必须上端到端模型 + 自建服务端。这是个很硬的技术判断。
3. <strong>长期记忆 = 必须搭服务端</strong>。现在大模型都是短期记忆，只记住最近几轮；想做"养成/陪伴"的长期记忆，必须服务端把这些东西存下来。"养成"不是模型送的，是工程堆的。
4. <strong>甲方的真实节奏是"先做样品 → 预售测市场 → 上量再补服务端"</strong>，不是一上来全做。乙方提醒：哪怕你预售只卖 20 个，服务端该有还得有，否则语音对话根本跑不起来——别把"以后再说"用错地方。
5. <strong>硬件方案要为"以后想加的功能"提前预留</strong>。板子一旦做出来、批量打样，后面想加陀螺仪/六轴/红外这些就来不及——接口和选型必须在打样前定死。这是硬件区别于软件最反直觉的一条。
6. <strong>离线 vs 海外，语言是隐形成本</strong>。想出海就只能押英文，国内押中文；离线指令一旦要做小语种，"很麻烦很搞"。出海野心要在硬件/离线方案阶段就算进去。
7. <strong>退货率是 AI 玩具行业的暗病</strong>。乙方点名"汕头澄海那堆只会接官方做个聊天就跑路的"，退货率本来就 60%——产品没有特征性、纯 to B 定制化，就是没竞争力。这是甲方想做"有差异化产品"的根本动机。

这场会练的是<strong>把一个模糊的"AI 玩具想法"翻译成技术可行性 + 成本 + 风险三张表</strong>的眼力。读完该记住三张图：①一条"能力归属判定线"——每个功能到底是模型能力、需要自建服务端、还是纯硬件，判错就报错价、踩错坑；②一条"产品节奏线"——样品→预售→上量→补服务端，钱往哪个阶段花；③一条"硬件不可逆约束线"——为什么硬件必须提前为未来预留。别记甲方想做什么玩具，记这套"拆需求"的方法。

## 📚 深挖：7 个值得单独学的知识点

### 1. "小智官方那套" vs "自建服务端"——AI 硬件最关键的一道分水岭
乙方反复在讲：小智（一个开源/平台化的 AI 硬件语音方案）官方那套，适合"一两个人用、快速低端上线"；但<strong>你只要商用、上量、要自定义，官方那边即使你只有一百台也不会让你用</strong>，必须自己搭云端去调外部大模型。判错这条，后果是报价、周期、能力边界全错。
- <strong>为什么重要</strong>：这是 AI 硬件"能不能做出差异化产品"的根命题。接官方=和汕头澄海一堆货同质化、退货 60%；自建=有特征性、能定制，但成本和工期完全不同。
- <strong>你能用</strong>：评估任何 AI 硬件项目，第一个动作就是判定"这是接官方平台，还是要自建服务端"，因为它决定后面所有的钱和坑。

### 2. 情绪感知为什么"小智那套天生做不了"——声学特征损失
这是全场最硬的一个技术点。乙方解释：小智那套链路是 VAD（人声检测）→ASR（语音转文字）→大模型→TTS（文字转语音）。<strong>语音先被转成文字，语调、语速这些"声学特征"在转文字那一刻就被丢掉了</strong>，所以从语气角度它根本识别不了你的情绪。要做情绪感知，只能用"端到端"的高级模型，且必须自建。
- <strong>为什么重要</strong>：很多人以为"情绪感知"是接个 API 就有的功能，其实它和整条技术架构绑定——选了 ASR 转文字的链路，就注定做不了。架构选择决定能力上限。
- <strong>你能用</strong>：凡是涉及"理解语气/情绪/声音特征"的需求，先确认链路是不是端到端，否则承诺了也做不出。

### 3. 长期记忆 = 服务端工程，不是模型自带
甲方把"长期记忆能力"列进需求，以为是模型能力。乙方纠正：现在所有大模型都是<strong>短期记忆</strong>，只记得最近几轮；想做长期记忆/养成，<strong>必须搭服务端把这些东西存下来</strong>。"养成"是工程堆出来的，不是模型送的。
- <strong>为什么重要</strong>：陪伴类产品的灵魂就是"它记得我"。但这恰恰是模型不提供、必须自己存储和管理的部分，是自建服务端的核心理由之一。
- <strong>你能用</strong>：任何"陪伴/养成/记得用户"的需求，直接对应到"需要服务端 + 数据存储"的成本，别当成免费的模型能力。

### 4. 产品节奏：先样品 → 预售测市场 → 上量再补服务端
甲方的思路很清晰：先做个样品宣传、做预售，预售可能就 20 个人买，项目没铺开；等超过某个量（比如 5000 人）再去考虑服务端搭建。乙方接：但服务端"该有的还得有"，没有服务端连语音对话都跑不起来——这是把"重资产投入往后放"和"基础设施不能省"两件事分开。
- <strong>为什么重要</strong>：这是硬件创业的现金流智慧——别在没验证市场前砸重资产，但也别把"必需基础设施"当成可省的东西混为一谈。
- <strong>你能用</strong>：做硬件/重资产产品，先用最小样品 + 预售验证真实需求，再决定要不要投服务端这类重投入。

### 5. 硬件的"不可逆约束"：未来功能必须打样前预留
乙方强调最值得硬件小白记住的一条：板子一旦做出来、批量打样，后面想加陀螺仪、六轴传感器、红外这些<strong>就来不及了</strong>——"几批样板一样的（白做）"。接口、选型、板子方案都必须按"未来想扩展什么"提前定死。
- <strong>为什么重要</strong>：软件可以迭代，硬件不行。这是硬件思维和软件思维最大的分野，也是新人最容易踩的坑。
- <strong>你能用</strong>：做硬件先列"未来 1-2 年可能想加的功能"，把对应传感器/接口在第一版板子就预留出来。

### 6. 社交功能怎么实现——红外/NFC 的具体取舍
甲方想要"两个娃娃靠近时能相互感知、打招呼、做表情互动"。乙方给的方案：红外可以做到"两机不用碰就能感应"；退而求其次可以用 NFC（需要更近距离）。晃动检测则用六轴传感器。这是把一个"产品想法"直接落到"用哪个元器件"的现场示范。
- <strong>为什么重要</strong>：它展示了产品需求→硬件元件的真实翻译过程，每个"感知/互动"需求背后都对应一个具体传感器和它的物理限制。
- <strong>你能用</strong>：把你的交互需求逐条问"靠哪个传感器实现、它的物理限制是什么"，别停在功能描述。

### 7. 表情/UI 受硬件性能反向约束——分辨率、帧率、内存
甲方想要"丰富彩色、好看的动态表情"。乙方泼冷水：表情再好看，<strong>屏幕分辨率、刷新率、内存不够都不行</strong>；要么用小屏，要么压缩图片（会损失质量）。所以必须先定"表情图片的最大分辨率、帧率、容量"，硬件方案才能跟着定。
- <strong>为什么重要</strong>：又一个"软件想法被硬件物理性能反向卡住"的例子。视觉效果不是设计说了算，是硬件性能说了算。
- <strong>你能用</strong>：做带屏硬件，先确定显示资产（图片/动画）的分辨率、帧率、容量上限，再谈视觉设计。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：每个需求先做"能力归属判定"（模型 / 服务端 / 硬件）</strong>
- 核心意思：拿到需求清单，逐条判定它靠模型能力、需要自建服务端、还是纯硬件——这决定报价、周期、可行性。
- 你可以怎么用：把你的产品需求做成三列表，每条标"谁来实现"，再去找方案商。
- 不要误读成：把所有"智能"功能都当成模型自带——情绪感知、长期记忆都不是。

<strong>方法 2：重资产分阶段投入（样品→预售→上量→补服务端）</strong>
- 核心意思：先用最小样品 + 预售验证市场，再投服务端这类重资产。
- 你可以怎么用：把你的产品拆成"验证市场所需的最小版本"和"上量后才投的部分"。
- 不要误读成：把"必需基础设施"（如能跑通对话的服务端）也当成可省的——该有的还得有，否则样品都跑不起来。

<strong>方法 3：硬件为未来预留，软件思维会害死你</strong>
- 核心意思：板子打样前必须把未来可能加的传感器/接口定死，硬件不可逆。
- 你可以怎么用：做第一版硬件前，先列未来想加的功能并预留接口。
- 不要误读成："先做出来再迭代"——这套软件逻辑在硬件上行不通。
