## ⏱ 如果只读 10 分钟（6 条带走判断）

1. <strong>乙方接单最大的坑是"需求没敲死就开工"</strong>。全场反复出现："要先跟客户理完所有需求才算开始做的时间，不然容易踩坑"；"总是被人家改需求，是因为我们没确认好"。这是这帮人花真金白银换来的教训，也是所有项目制乙方的命门。
2. <strong>签合同是"改需求"的分水岭，要在签约前就把规则讲清楚</strong>。共识：没签合同没打款前，客户随便改、我们只对接信息不开工；一旦合同写死了功能/指标，再改就动交期和费用，且"这些都会写在合同上"。把改需求的成本前置写进合同，是防扯皮的核心动作。
3. <strong>用"需求模板"把销售对接标准化</strong>。他们要做一张表/模板，现场拿去逐项问客户（外观造型？显示还是语音还是多模态？几个维度动作？传感器/红外/NFC/雷达？），"让他讲，我往里面填就行"。把"会聊天才能接单"变成"照表问就能接单"。
4. <strong>硬件做成"可复用模块"，把项目利润摊薄成边际成本</strong>。计划做一块"什么接口都引出来"的开发板/母板，后面接新单就"扣哪个模块拿哪个模块"，改个版型即可——"后期再接，几十块钱成本就吸收了"。这是乙方从"每单重做"走向"攒资产"的关键一步。
5. <strong>客户质量靠"聊"筛，不靠流程判断</strong>。原话："很多都是聊的时候才能看出来。"实干型创业者（愿意线下见、愿意投钱、意愿强）是优质客户；犹豫不决、反复要反馈又说不清想干嘛的，要设次数上限。
6. <strong>基建（共享资料库 + 项目时间线）要趁早、但别过度</strong>。先用微盘/云盘丢"我们能做什么 + 成熟硬件方案"，方便手机随时看；再做一张项目时间线表，好让"等结构出图的空档去推进别的单"，同段时间并行接好几个项目。

## 📚 深挖：7 个值得单独学的知识点

### 1. 项目制乙方的头号死因：需求没敲死就开工（这是全场的"道"）
全程出现至少三次同一个痛点："要先跟客户理完所有需求之后才算开始做的时间，不然容易踩坑"；"为什么总是被人家改需求，是因为我们没确认好这个"；"我也不太懂这一块，踩雷算了，这个经验真的是。" 这是带头人自己承认踩过的雷。底层逻辑：乙方的成本是工时，需求一变工时就重置，而口头需求天然会漂移。<strong>你能用</strong>：任何项目制交付，把"需求确认完成"定义成"项目开始计时"的那条线，确认前不开工。

### 2. 合同节点 = 改需求的分水岭（这是最实的"法"）
他们梳理出一套很清楚的规则：<strong>没签合同、没打款之前</strong>，客户随便改，"我们只是给他对接信息，这一阶段他是可以改的"，不开展实际工作；<strong>一旦签了合同</strong>，把"做哪些功能、做到什么指标"敲定写进合同，"后期再要更改，交期和费用可能有变动，这些要提前跟他说，都会写在合同上"。关键是<strong>前置沟通</strong>——签约前就把"以后改要加钱加期"讲明白。<strong>你能用</strong>：在你的合同/报价里写死两段式改动规则，并在签约前口头确认一遍，把扯皮成本前移。

### 3. 用"需求模板"把销售对接从手艺变成流程
带头人自称不会聊、技术班子也不擅长对客户，解法是做一张需求模板表："现做一个模板，后面不管现场哪个客户，我们就找一个模板去问他，让他讲需求，我往里面填就行了。" 要问的维度已经列出来了：外观造型有没有/复杂度、内部要什么功能（显示/语音/多模态）、几个维度动作、传感器（红外/NFC/雷达/姿态感应）。甚至想到"直接去找同行要现成模板"。<strong>你能用</strong>：把你最依赖"老销售手感"的环节做成一张采集清单，让任何人照着问都能收齐需求。

### 4. 硬件模块化母板：把一次性项目摊成边际成本
计划做一块"全套的板，各种接口都引出来"的开发板，"先收集现有几个项目都具备哪些功能，全部集成在一个板块"，相当于一块母板。后面接新单"你想用哪个模块就扣哪个模块进去，改一下版本就行"。算的账："后期再接这样的单，几十块钱成本就吸收了。" 痛点也点了：电路好模块化，<strong>结构难定</strong>（因为外观随客户变）。<strong>你能用</strong>：把你重复交付里"每次都重做"的部分抽成可插拔模块，第一次贵、后面接近零成本——这是乙方攒资产的核心动作。

### 5. 两阶段出方案：别在签约前免费给详细方案
明确的策略：客户提需求时（要多模态、传感器、几个维度动作），"我们都很熟悉了，但不给他太详细的方法；签了合同我们再出详细方案"。签约前只给"简单/模糊的框架"，详细建模、详细方案留到签约后——因为"建模比较费时间，要算工期"。<strong>你能用</strong>：把"详细方案/详细建模"当成签约后才交付的工作量，签约前只给框架，避免免费做白工被白嫖。

### 6. 客户质量靠"聊"筛，并给犹豫客户设次数上限
判断："很多都是聊的时候才能看出来，没办法说一下就确定。" 优质客户的画像很具体：实干、愿意线下见面、上次聊就看得出想把事做好、愿意投钱、意愿强烈（如"林总""一朵"这类）。反面："犹犹豫豫都不知道他想干什么"的客户，要设反馈次数上限（"最多反馈几次"）。还有个现实风险提示："有的到签合同了，公司又取消项目，没办法。"<strong>你能用</strong>：建立一个轻量的客户分级判断（意愿/实干/投入度），对低质客户限制免费对接的次数和深度。

### 7. 固件↔后端通讯：从"单向请求"到 WebSocket 双向
一段真实的技术对齐：现状是"硬件向我（后端）发请求，我返回给你"的单向模式；但需求是后端要能<strong>主动下发动作</strong>（比如模型解析出"点头"，要把"点头"指令下发到固件去执行）。后端的人坦言"我没这么做过"，但判断"用 WebSocket 协议能做，因为双向都能发"。他们硬件侧用的疑似是小智系的协议（ASR 识别为"扫地协议"，存疑），方案锁定 ESP32（换其他 MCU 要重新开发、周期长）。<strong>你能用</strong>：需要服务端主动控制设备（而非设备轮询）时，WebSocket 这类长连接双向协议是默认解；技术选型一旦成熟（如 ESP32）就别轻易换，沉没的是开发周期。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：需求确认 = 项目开始计时线</strong>
- 核心意思：把"需求理清"定义成项目正式开工的起点，确认前不投入实际工时。
- 你可以怎么用：在流程里加一道"需求确认签字/确认"关卡，过了才排期。
- 不要误读成：要把需求一次性问到 100% 完美——而是把"已确认范围"和"未确认/后续变更"清晰切开，后者走变更流程。

<strong>方法 2：两段式改需求规则写进合同</strong>
- 核心意思：签约前随便改、不开工；签约后改动交期与费用，且签约前就讲明白。
- 你可以怎么用：合同/报价模板里写死这两段规则，签约前口头复述一遍。
- 不要误读成：签约后就拒绝一切改动——是改动要计价、要重新约定交期，不是不让改。

<strong>方法 3：把重复交付物模块化攒成资产</strong>
- 核心意思：第一次把可复用部分（开发板/模块/资料库）做出来，后面接单靠拼装。
- 你可以怎么用：盘点你"每单都重做"的部分，第一次多花点成本做成可插拔模块。
- 不要误读成：所有东西都能模块化——结构/外观这类随客户强定制的部分难复用，别硬塞。
