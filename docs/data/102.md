## 📚 深挖：6 个值得单独学的知识点

### 1. 知识产权 ≠ 源代码：把资产和实现分开谈
现场甲方一开始根本没盯知识产权，是"最后才看，发现知识产权全归乙方"才炸。谈到最后理清了三层：<strong>软件著作权/设计权/算法安全模型权归甲方</strong>（甲方要拿去申请专利、对外销售必须有这个）；<strong>源代码留乙方</strong>（甲方主动放弃，承诺不反编译、不逆向工程）；甲方还承诺"要新增功能还来找你开发"。
- <strong>为什么重要</strong>：很多人把"知识产权"和"源代码"当一回事，结果要么把命根子（著作权/专利权）拱手让人，要么纠结一个自己根本用不上的源码。两者完全可以拆开：你拥有"对外销售和申请专利的权利"，对方握有"实现细节"，互不耽误。
- <strong>你能用</strong>：签外包合同前，先把"知识产权归属"和"源代码交付"列成两行分别谈，别打包。

### 2. "第三方 SDK"类比：判断上层 IP 该归谁的尺子
甲方原话大意："我用第三方视频 SDK 做了个能视频聊天的 App，这 App 的知识产权当然归我，但我没有你这个 SDK 也动不了。" 他用这个把乙方"上层场景也归乙方"的要求顶了回去。
- <strong>为什么重要</strong>：这是一条可迁移的判断规则——<strong>底层引擎/能力的 IP 归提供方，基于它搭建的业务/场景/品牌 IP 归甲方</strong>。它既划清了边界，又给了对方台阶（你的底层我不碰、不反编译）。
- <strong>你能用</strong>：任何"我基于你的能力做了我的产品"的合作，都可以用这个类比快速划 IP 边界。

### 3. "两头钱不能都吃"：用对称性谈分成
甲方反复讲的公平逻辑：如果乙方要硬件分成，那就是"大家一起做"，源代码和知识产权该甲乙共有；如果乙方要独占源代码，那分成就得去掉。"你这头吃了，那头你要让出来。"最终成交：<strong>乙方去掉提成、守住源代码</strong>。
- <strong>为什么重要</strong>：这是谈判里很硬的对称性原则——<strong>对方在一个维度多拿，就必须在另一个维度让步</strong>，避免对方"既要分成、又要独占 IP、又不给源码"的三头通吃。
- <strong>你能用</strong>：谈合作时把对方的诉求一条条摆出来，用"那你在另一项上让什么"来对冲，逼出对称结构。

### 4. 不要源码可以，但要"能自己运维"的三件套
甲方放弃源代码的同时，死咬要：<strong>部署脚本、开发环境、技术文档</strong>。理由："我可能请一两个运维；你连怎么部署服务器都不告诉我，合作没法长久。"
- <strong>为什么重要</strong>：源代码是"能改它"，运维三件套是"能跑它、迁它、不被锁死"。放弃前者不等于接受一个完全黑盒——否则甲方会被乙方在运维上长期卡脖子。
- <strong>你能用</strong>：外包交付清单里，除了功能，单列"部署脚本 + 开发环境 + 技术文档"为必交项。

### 5. 硬件项目的成本链：BOM 决定生死，不是功能
甲方算的是一条利润链：乙方收费 → 经销商加利润 → 自己只剩约 3% 毛利 → "结果都是亏的，我牵头做干嘛"。所以他要在 license 上"起码扣 20 块下来"，硬件"做到两三百就卖不出去"。这也是他突然要换 API 的根本动机——压成本。
- <strong>为什么重要</strong>：硬件不像软件，每一分 BOM 成本都乘以销量，还要给渠道留利润空间。<strong>功能再好，成本压不下来就没有销量，所有营销/代言都白做。</strong>
- <strong>你能用</strong>：硬件立项第一件事是倒推 BOM 和分润链，确认终端价能跑通毛利，再谈功能堆叠。

### 6. AI API 选型：价格、客户归属、工程改动三角权衡
后半程的核心是从百度切到小智或青禹：
- <strong>百度</strong>：一个 license 约 40，"太贵"，是被替换的对象。
- <strong>小智 Xiaozhi</strong>：按 license 约 22、量大可再降；提供语音对话 + 视频 + 大模型万物识别接口；官方明确支持你在自己 PC/小程序/APP 上调 API 管理设备和用户，<strong>不连它的平台就能"无限用 token"，客户归你</strong>；连了平台则客户被共享。
- <strong>青禹AI</strong>：约 20，更便宜，基于杰里 AC792 芯片、自带 SDK（拿来即用更简单），疑似也支持 ESP32——但换它意味着<strong>复印件/硬件/固件全部重改，BOM 要重算</strong>。
- <strong>为什么重要</strong>：选 API 表面比单价，实际在三件事之间权衡——<strong>单价成本、用户数据归谁、换方案带来的工程返工量</strong>。便宜的方案可能让你整套硬件重做。
- <strong>你能用</strong>：选型时把"价格/客户归属/改动成本"三列摆出来打分；并坚持"商务+技术拉群对齐"，别只听一方报价就拍板。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：把外包资产拆成"IP / 源码 / 分成"三个独立筹码</strong>
- 核心意思：三者分开谈、互为对价，而不是打包成一句"知识产权归谁"。
- 你可以怎么用：合同里分三节写——著作权与专利权归属、源代码与运维交付、硬件分成比例。
- 不要误读成：把源代码当成必争之物——甲方这场恰恰主动放弃了源码，只守住了著作权和可运维性。

<strong>方法 2：用"第三方 SDK"类比划 IP 边界</strong>
- 核心意思：底层能力 IP 归提供方，上层业务/品牌 IP 归使用方。
- 你可以怎么用：被对方过度索权时，举一个对方也会用的生活化类比把边界讲清。
- 不要误读成：所有东西都归甲方——甲方明确承认底层引擎/源码是乙方的、不碰不反编译。

<strong>方法 3：换 API 前先做"全链路改动评估"</strong>
- 核心意思：换服务端/换芯片不是改一行配置，是固件、硬件、复印件一起重做。
- 你可以怎么用：换方案前，让技术评估改动范围 + 重算 BOM，再决定值不值。
- 不要误读成：便宜 20 块就一定换——返工成本和上市延期可能吃掉省下的钱。
