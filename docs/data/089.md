## ⏱ 如果只读 10 分钟（7 条带走判断）

1. <strong>AI 硬件的体验瓶颈，八成卡在"收音"这一关，不在大模型。</strong> 整场最反复的结论：唤醒不灵、识别不准、延迟高、不响应——根因大概率是麦克风收到的人声太小、被机器自己的播报回声/共振噪声掩盖，后台识别不出来就不响应。软件能调灵敏度，但"硬件不先解决，软件白调"。

2. <strong>不装壳一切正常，装壳就出问题——壳本身就是一个噪声源。</strong> 裸板测试收音正常，装上外壳后机器自身扬声器的振动通过外壳共振，被麦克风收回去形成回声/杂音。解决靠"塞东西"改变空腔共振频点 + 麦克风和扬声器两处都垫软硅胶（30度）做声学隔离，避免硬接触一起震。

3. <strong>方案商最痛的不是技术，是"边界"。</strong> 客户事后冒出来的需求（充电指示灯、删按钮、按键不凹进、灯带改霓虹、闹钟、知识库优先级、方言输出……），逐条要先判"这是谁的责任、要不要收钱"。判据是一条铁律：<strong>当初有没有发图/发结构图给客户确认过</strong>——发过且客户认了的，是客户二次需求要收费；没发清楚的，是己方沟通锅，认改。

4. <strong>改一次图、收一次费，但收费要"打包进第一点"，别零敲碎打。</strong> 老板的报价心法：外观/结构相关的所有新增费用，统一并到"第一点（改图重打样）"里报，别一个点一个点加钱——客户会觉得"这也加那也加"而反感。

5. <strong>豆包大模型这一层就有 ~2 秒固有延迟，链路要换更快的组合。</strong> 现场测出豆包慢（隔 2 秒，极端到 11 秒），最快路线被指向"豆包 ASR + 豆包 TTS + 某天文/先锋方案"约 1 秒一轮。延迟问题要先排除"是不是网络/是不是没收到声音"，再归因到模型。

6. <strong>声音克隆 + 唤醒加速会"缓存第一句"，这是个隐藏 bug 也是性能特性。</strong> 选好音色后，系统会用该音色预缓存一条打招呼音频；下次唤醒直接播缓存、不走大模型，所以快——但若没做好，第一句会回放成原始音色而非克隆音色。嫌麻烦可直接关掉该加速功能。

7. <strong>"客户不会用 ≠ 产品没做完"，但方案商得替客户兜底到能跑通。</strong> 多个功能（MCP 接入点、钉钉、声纹识别）其实"本来就支持，调一下服务器 ID 到 MCP 接入点就行"，但客户不会部署。结论：得把 MCP 配好 + 写一个示例给客户，否则"支持"对客户等于"不存在"。

## 📚 深挖：7 个值得单独学的知识点

### 1. 收音是 AI 语音硬件的"第一性问题"
<strong>是什么</strong>：全场把"唤醒不灵、命令词要喊几遍、识别准确率低、延迟高、不响应"反复归到同一根因——麦克风收到的人声太小，且被机器自己播报的声音/外壳共振产生的回声和杂音掩盖。后台日志里"豆包（机器播报）声音很大、人声很小"，解析不出语句自然不响应。现场原话量级的判断：识别不出来"会被噪声掩盖，它就不响应"。
<strong>为什么重要</strong>：很多团队遇到"AI 不聪明"，第一反应是换更大模型或调 prompt，但在硬件上，输入信号（人声）质量才是上限。垃圾进、垃圾出——再强的 ASR/LLM 也救不了一段被回声污染、人声极弱的音频。这是可迁移到一切语音/音频/传感硬件的底层规律。
<strong>你能用</strong>：体验差先做信噪比排查（后台听原始音频里人声 vs 噪声比例），把"收音"当第一嫌疑人，而不是模型。

### 2. "不装壳正常、装壳就坏"——外壳是声学系统的一部分
<strong>是什么</strong>：裸板内测收音正常，装壳后出问题。机制：扬声器播报时振动通过外壳形成空腔共振，被麦克风收回，叠成回声/杂音。两个解法并用——①往内部空腔"塞东西"改变共振频点，让回声没那么大；②麦克风和扬声器<strong>两处都垫 30 度软硅胶</strong>做减振隔离，避免硬接触带着外壳一起震。麦克风放顶部弧面、与外表面留距离也是为此。
<strong>为什么重要</strong>：这把"结构设计"和"算法表现"直接绑死了——结构没做声学隔离，AI 表现就崩。它解释了为什么实验室 demo 好、量产体验差。是硬件团队软硬协同的经典必修课。
<strong>你能用</strong>：任何带"边播报边收音"的设备（音箱/对讲/陪伴机器人），结构阶段就要做扬声器-麦克风的减振隔离与回声路径设计，别等软件背锅。

### 3. 方案商定责的"发图确认"铁律
<strong>是什么</strong>：面对客户一长串"我要加这个、改那个"，老板的处理范式是逐条问两件事：当初<strong>有没有发过图/结构图给客户、客户认没认</strong>。发过且客户确认过的（如只有一个按钮的结构图），现在改属于客户二次需求→收费；没发清楚、没说明白的（如充电口/指示灯没体现），算己方沟通问题→认改不收费。整场反复出现"这个图是发给他看过的""他第一次发的图里只有一个按钮所以他以为是这样"。
<strong>为什么重要</strong>：这是甲乙方协作里最普遍的纠纷源。把"确认留痕"作为定责依据，既是收费正当性的来源，也是免责的护身符。可迁移到一切外包/定制/服务交付。
<strong>你能用</strong>：把"每个关键设计节点发图/发文档让客户书面确认"做成硬流程，留痕即边界。

### 4. 新增费用"打包进第一点"的报价心法
<strong>是什么</strong>：所有跟外观/结构相关的新增费用（重新改图、重新打样、模具/打样厂费用），统一并入"第一点"报给客户，而不是一个点一个点单独加价。理由是老板原话意思："一个点也加钱、那个点也加钱，客户会比较反感。"
<strong>为什么重要</strong>：这是报价的心理学——总价一次说清，比 N 次零敲碎打更易被接受，也减少每个点都讨价还价的拉锯。可迁移到任何项目变更管理。
<strong>你能用</strong>：变更收费做"合并报价、一次沟通"，别让客户感觉处处加钱。

### 5. 端侧语音链路的延迟从哪来：模型 vs 网络 vs 收音
<strong>是什么</strong>：客户反馈延迟高（有时 11 秒）。现场拆解：豆包大模型本身就慢（每跳约 2 秒），而 11 秒"不理解，7 秒应该是没收到声音"——即超长延迟更可能是没收到声音/解析不出而非纯模型慢。最快路线被指向"豆包 ASR + 豆包 TTS + 某天文方案"约 1 秒/轮（"先锋"那个更快的方案整流程约 1 秒，存疑命名）。还要排除网络问题。
<strong>为什么重要</strong>：延迟是端侧语音体验的命门，但它是多段叠加（收音→ASR→LLM→TTS→播放），定位要按链路分段，而不是笼统说"AI 慢"。
<strong>你能用</strong>：遇到语音延迟，按"收音/网络/ASR/LLM/TTS"分段打时间戳定位瓶颈；模型层优先测低延迟 ASR+TTS 组合。

### 6. 声音克隆 + 唤醒加速 = 预缓存第一句（含一个 bug）
<strong>是什么</strong>：选定音色后，系统在最后一次对话时用该音色<strong>预缓存一条"打招呼"音频</strong>；下次唤醒直接播这条缓存、不走大模型，所以唤醒回应快。但"克隆/加速没做好"时，第一句回应会变成<strong>原始音色而非克隆音色</strong>。嫌麻烦可直接关掉这个加速功能（关掉就不缓存、第一句也走正常链路）。
<strong>为什么重要</strong>：这是"性能优化引入正确性 bug"的典型——缓存提速的同时埋了"缓存内容用错音色"的坑。理解缓存边界才能既快又对。
<strong>你能用</strong>：做语音/对话缓存时，明确缓存键要带上"音色/人格"维度，避免快了却播错声音；不需要的优化先关掉降复杂度。

### 7. 方言：ASR 能识别、TTS 输出才是难点
<strong>是什么</strong>：客户想要"方言对话"。技术现状：ASR/TTS 分阶段做，<strong>输入端支持方言识别没问题，输出端不支持自动方言</strong>——要方言输出得在小程序侧手动选语言，或在 TTS 阶段自动判别加工具（成本上去、"太麻烦了"）。但现场又被豆包"同一音色能输出不同方言（如邻家女孩音色说四川话）"震到，怀疑豆包在 ASR 阶段识别到粤语后，在合成阶段也能带出来，"这是需要去找到的"。
<strong>为什么重要</strong>：揭示了语音产品里"听懂"和"用方言说出来"是两件难度不对称的事；也提示对标豆包这类已实现能力，去逆向人家的 pipeline。
<strong>你能用</strong>：做多语言/方言语音产品，分别评估 ASR 与 TTS 两端能力，别假设"能听懂就能用同口音回"；对标已实现的竞品定位差距。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：体验问题先查物理层（收音/结构），再查软件</strong>
- 核心意思：AI 硬件不灵，先怀疑麦克风收音和外壳共振，而不是模型。
- 你可以怎么用：后台调原始音频听人声/噪声比，结构上垫软硅胶、堵空腔改频点。
- 不要误读成：换更大模型/调灵敏度就能救——灵敏度调高会带来误识别，信号源不解决等于白调。

<strong>方法 2：用"发图确认过没有"给客户需求定责</strong>
- 核心意思：留痕即边界，确认过的改动收费、没说清的认改。
- 你可以怎么用：每个设计节点发图让客户书面确认，作为收费/免责依据。
- 不要误读成：所有客户新需求都收费——没发清楚的部分是己方沟通锅，硬收会失客户信任。

<strong>方法 3：变更费用合并进主项一次报</strong>
- 核心意思：所有外观/结构新增费用打包进"第一点"一次沟通。
- 你可以怎么用：变更管理做合并报价，减少逐点议价。
- 不要误读成：把杂费藏起来不透明——是合并沟通，不是隐瞒，客户问到要能拆得清。

<strong>方法 4：交付以"客户能自己跑通"为准，配置+示例一起给</strong>
- 核心意思："代码支持"不等于"客户能用"，要把 MCP 接入点配好并写示例。
- 你可以怎么用：交付每个功能附最小可运行示例 + 必要的服务器/接入点配置。
- 不要误读成：客户不会用是客户的事——对方案商而言，跑不通就等于没交付。

## 📅 7 天作业

- <strong>Day 1</strong>：把这台机现存问题按"硬件能解/软件能解"分两栏，先锁定收音类问题归硬件优先。
- <strong>Day 2</strong>：后台导出一段真实对话原始音频，量化人声 vs 机器播报/噪声的强弱比，确认收音是不是主因。
- <strong>Day 3</strong>：在麦克风和扬声器两处加 30 度软硅胶减振 + 往空腔塞物试改共振频点，对比装壳前后识别率。
- <strong>Day 4</strong>：给语音链路每段（收音/ASR/LLM/TTS/播放）打时间戳，定位 11 秒延迟到底卡在哪段。
- <strong>Day 5</strong>：测"豆包ASR+豆包TTS"低延迟组合 vs 现方案，记录单轮耗时。
- <strong>Day 6</strong>：复盘客户全部新增需求，逐条标"发过图确认/没发清楚"，产出收费清单合并进主报价。
- <strong>Day 7</strong>：把 MCP 接入点配好并写一份给客户的最小示例，验证客户能否照着自己跑通（钉钉/声纹/知识库其一）。
