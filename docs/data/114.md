> 场景：一场甲方（产品方/老板代表）和乙方（方案商/开发团队，含"胡工"）的项目验收拉扯会，约 29 分钟。核心争议：一个 AI 语音硬件（小智二开方案）的「音量/语速等高级设置」和「声音克隆 CosVoice」两个功能做不出来，项目已拖近 4 个月，双方在「这功能到底在不在合同里」「年前能不能交付」上反复拉锯，最后谈成一个 2 月 7-10 号交接的妥协方案。
>
>
> ⚠️ 本场信息密度偏低，且是 ASR 逐字稿、技术名词错得很多（"伊斯克隆 code voice"=声音克隆/CosVoice，"DTS"大概率是 TTS，"A token"=AI token，"职工/恐龙"等为识别噪声）。这不是一场"学知识"的分享会，是一场<strong>真实的甲乙方交付谈判</strong>——它的学习价值在「人情世故 + 项目管理」，不在技术细节。

## ⏱ 如果只读 10 分钟（6 条带走判断）

1. <strong>"做不了"要早说，不能拖 20 天再说</strong>。乙方调了 20 天那个高级设置功能，最后一句"做不了"，甲方当场怼："做不了你早点说。" 拖延的代价不是技术债，是信任。
2. <strong>架构选错了层级，功能就从"小改"变成"重写"</strong>。乙方一开始把所有参数设成<strong>全局/管理员级别</strong>，现在甲方要的是<strong>智能体/设备级别</strong>——这俩不是改个参数的事，是要"一层一层往下传递"，等于动底层。设计之初的层级选择，决定了后期改动是几行代码还是推倒重来。
3. <strong>合同写得模糊，争议时谁都说不清</strong>。甲方说"功能明确写在合同里"，乙方反问"你看合同写得多模糊，按什么什么小字"。整场最大的内耗来自需求边界没在合同里钉死。
4. <strong>"和小智一样"这种口头需求是埋雷</strong>。甲方当初说"跟小智一样"，乙方据此没做支付/某些模块，后期甲方又说要——"和某某一样"是最不可执行的需求描述。
5. <strong>硬件能解决的，别硬塞软件</strong>。乙方反复主张：音量/语速直接对着设备说"声音大一点"就行（硬件层已实现），没必要在软件端做一套设备级控制。这是"用对的层解决问题"的工程判断，也是甲方需求和工程现实的拉扯点。
6. <strong>量产成本是另一本账，和打样完全两回事</strong>。乙方点破：现在这套方案 token 消耗成本高、量产很贵；真正量产的两单都改用了<strong>百度</strong>的服务端（语音可控到 18，比声网/火山/小智官方便宜）——但因为合同要求"小智二开"，这个项目反而绑死在贵方案上。

## 📚 深挖：5 个值得单独学的知识点

### 1. 全局级 vs 设备级 vs 中间级——架构层级决定改动成本
现场反复在讲一件事：原方案把 TTS 的音色/语速/角色等参数做成<strong>全局</strong>（管理员在后台统一设置，类似"改个 API Key 那种全局配置"），所有设备一改全改。但甲方要的是按<strong>智能体/设备</strong>粒度去设——"一个智能体可以对多个设备"。乙方说得很实在："最高层或最低层都好改，要改在中间，你就得一层一层去传递……每次请求还得去搜这个用户的信息、智能体的信息，再拼起来组成请求。"
<strong>为什么重要</strong>：这是所有带"作用域/权限"系统的通用规律。全局配置最省事但最不灵活；下沉到设备/用户粒度，意味着每次请求都要做归属查询和数据拼装，逻辑复杂度指数上升。这个层级是<strong>设计之初就定死的</strong>，后期想改就是动底层。
<strong>你能用</strong>：做任何"设置/偏好"类功能前，先画清楚作用域树（全局→分组→设备/用户），早期就定准粒度，别等做到一半发现要"从全局改成设备级"。

### 2. "和 XX 一样"是需求描述里最大的雷
甲方当初的需求是"跟小智一样"。结果：乙方据此没做支付模块、按全局模式做了参数控制，到后期甲方说"还要写支付""要设备级"，乙方反问"合同里哪有支付，你们说跟小智一样"。
<strong>为什么重要</strong>："和某竞品一样"听起来清楚，实则把一个黑盒丢给对方——对方对这句话的理解和你脑子里的画面几乎不可能一致。它是甲乙方扯皮的头号源头。
<strong>你能用</strong>：无论甲方还是乙方，把"和 XX 一样"翻译成<strong>具体功能清单 + 验收标准</strong>再签字，否则这句话后期一定回来咬你。

### 3. 打样能跑 ≠ 量产可控——成本是另一本账
乙方点破一个甲方似乎没意识到的事："你们现在这个版本是量产不了的，量产代价很高，token 消耗成本都很高，你测一下就知道。" 并给了一个毛利测算逻辑：给到销售端最少要砍一半（给 600，自己得能卖 300 出去），那 token / API / license 成本就得压到很低。
<strong>为什么重要</strong>：很多硬件项目打样阶段功能跑通就以为成了，但量产时单位成本（尤其 AI 推理/TTS 的 token 成本）会把毛利吃光。这是 AI 硬件特有的"打样陷阱"。
<strong>你能用</strong>：AI 硬件立项就要做<strong>单台量产成本测算</strong>（token + API + license），并和"能卖多少钱"倒推毛利，别等量产才发现亏本。

### 4. TTS 服务商横评：百度最便宜，小智官方押金重
乙方说他们真正量产的两单都没用小智方案，而是测了<strong>声网、火山引擎、百度、深网</strong>四家，<strong>百度最便宜</strong>——"只要语音的话可以控制在 18"。小智官方则是"20 万押金，每台 20 块"。本项目之所以贵，是因为合同要求"小智二开代码"，协议不一样，没法换百度。
<strong>为什么重要</strong>：这是一份真实的、带价格的服务商对比情报（虽是销售口径需核实）。也说明<strong>技术选型被合同条款锁死</strong>会直接抬高成本。
<strong>你能用</strong>：做语音硬件先横评主流 TTS 服务商的量产单价和押金模式；签合同时别把自己锁死在某个特定技术栈（如"必须二开 X"）上，留切换空间。
>
>
> ⚠️ "18""20万押金/每台20""声网/火山/百度/深网"均为现场口径与 ASR 识别，价格和厂商名（"深网"存疑，可能是"声网"或别的）需独立核实。

### 5. 夹在中间的人怎么把僵局谈成交付
最后 1/3 是一段教科书式的收尾谈判，关键动作链：
- <strong>先共情泄压</strong>："我也比较累，我夹在中间，老板那边给我压力也挺大，互相理解一下。"
- <strong>给硬截止</strong>：老板要 2 月 10 号前全部交接，自己内部目标提前到 <strong>2 月 7 号</strong>，留 3 天交接缓冲。
- <strong>锁死范围</strong>：明确只剩 CosVoice 一个功能；"流程级的大改动肯定不会再搞"，"别再弄一个大大的长列表来"。
- <strong>设妥协退路</strong>：音调/语速那个高级设置"我去跟老板沟通走统一管理行不行"——把做不出来的功能往"老板能接受的简化版"上引。
- <strong>绑定付款</strong>："东西弄好之后把钱打了就 OK"——交付和回款挂钩。
<strong>为什么重要</strong>：这是真实商业里"项目要黄但还得收尾"时的标准操作，比任何谈判书都具体。
<strong>你能用</strong>：项目陷入拉锯时，用「共情 + 硬截止 + 锁范围 + 给退路 + 绑回款」五步把无限拖延逼成有限交付。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：动手前先定"作用域粒度"</strong>
- 核心意思：全局 / 分组 / 设备-用户级，三种粒度改动成本天差地别，出生就要选对。
- 你可以怎么用：新功能设计先画作用域树，确认粒度后再写代码。
- 不要误读成：永远做设备级才灵活——粒度越细，每次请求的归属查询和数据拼装成本越高，按真实需求选，别过度设计。

<strong>方法 2：把"和 XX 一样"翻译成清单再签字</strong>
- 核心意思：竞品对标不是需求，功能清单+验收标准才是。
- 你可以怎么用：收到"和小智一样"立刻追问"具体要哪几个功能、怎么算验收"，写进合同。
- 不要误读成：把合同写厚就安全——关键是把<strong>会扯皮的边界点</strong>（支付、权限粒度、克隆）逐条钉死，而不是堆字数。

<strong>方法 3：收尾用"硬截止 + 锁范围"</strong>
- 核心意思：项目要黄时，靠截止日和范围锁定逼出有限交付。
- 你可以怎么用：给一个明确日期 + 一句"流程级大改不再接" + 留缓冲。
- 不要误读成：锁范围=甩锅——共情和退路（简化版方案）要同时给，否则关系直接崩、回款也没了。
