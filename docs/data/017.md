## 📚 深挖：5 个值得单独学的知识点

### 1. GPS PPS 授时：1ms 对时是怎么做到的（全场的"道"）
甲方讲得很清楚：GPS 模块会输出一个 <strong>PPS（Pulse Per Second，每秒脉冲）信号</strong>，是一个上升沿；PPS 上升沿到来后<strong>过几百毫秒</strong>才收到一条报文，报文里含"秒数、日期"等时间信息——<strong>报文里的时间对应的是那个上升沿的时刻</strong>。做法：拿 PPS 上升沿做硬件触发/打时间戳，再用随后报文里的绝对时间去标定，从而把整个单片机时钟同步到 GPS。目的是"给采集数据的时间戳对准，以 GPS 时钟确定采集时间"，要求误差 <strong>1ms 以内</strong>。
<strong>为什么重要</strong>：这是所有需要"多设备时间对齐"场景（地震/振动监测、电网同步、分布式传感阵列）的通用手法——不靠 NTP，靠 GPS PPS 拿到纳秒级稳定的秒沿。理解"上升沿给精度、报文给绝对时间"这层分工，是做授时类产品的入门关。
<strong>你能用</strong>：要做高精度时间戳的数采，优先 GPS PPS 方案；把 PPS 接到 MCU 的输入捕获/外部中断引脚，报文走串口解析，二者结合。

### 2. 把整台设备读成一张"低功耗采集节点"状态机
老手当场口述了完整流程，值得抄成状态机：<strong>定时上电/唤醒</strong>（写死固定时间，如每晚 8 点，或用 RTC 唤醒）→ <strong>找 GPS 信号</strong>（找不到就直接休眠，省电）→ <strong>PPS+报文对时</strong>（同步到 1ms）→ <strong>采 1 分钟数据</strong> → <strong>本地处理</strong>（这段甲方自己回填算法，外包方只需"把口子留出来"）→ <strong>判断是否报警/上报</strong>，需要就经蓝牙/4G/NB 回传 → <strong>休眠</strong>等下次唤醒。Flash 只存一天数据，次日可覆盖。
<strong>为什么重要</strong>：这是电池供电野外节点的范式——绝大部分时间在睡，醒来只做最小必要工作。"找不到信号就休眠"这种细节，正是把功耗压下来的关键决策。
<strong>你能用</strong>：设计任何电池供电采集设备，先画"睡眠为默认、唤醒做最小事"的状态机，并明确每个外设关电的时机。

### 3. "三选一回传"与外设模组化：陌生硬件没那么可怕
回传通道是<strong>蓝牙 / 4G / NB-IoT 三选一</strong>（同一套串口，运行时三选一，不全删）。甲方反复强调：GPS、蓝牙、4G/NB 这些"实际上都是把一个模组贴上去的"，对单片机而言"就是串口通信"——蜂窝/蓝牙模组走 <strong>AT 指令</strong>，GPS 走<strong>标准报文</strong>，"不用太关注芯片的具体工作型号"。
<strong>为什么重要</strong>：这点解释了为什么一个没见过这块板子的人敢接单——只要外设是"模组+串口"，复杂度就被封装掉了，工作量从"驱动十几颗陌生芯片"降为"写几路串口协议解析"。这是嵌入式外包能跨硬件接活的底层原因。
<strong>你能用</strong>：接陌生硬件先盘点外设是不是模组化（AT/标准报文/串口）；是，就敢接，按串口协议逐个对即可。

### 4. 采样频率 vs 通信速率：一次算账消除技术焦虑
甲方反复担心"单片机采样频率太快、数据速率跟不上"，把<strong>采样频率</strong>和 <strong>SPI 通信速率</strong>混在一起。老手用一次算账拆穿：ADC 是<strong>独立芯片</strong>（不是 MCU 内置 ADC），位宽 24bit，按 32bit 传输算；即便 <strong>10K 采样/秒 × 32bit = 320Kbps</strong>，而 STM32 的 SPI 轻松跑到 <strong>5-6MHz</strong>——"套了三个数量级，怎么可能达不到"。结论：用 <strong>DMA 或中断</strong>收，完全没问题。（甲方提到目标采样率 20K，可降到 10K 以下，"通常想做 20K"。）
<strong>为什么重要</strong>：这是工程沟通里极常见的认知错位——把"每秒采多少个点"和"总线能传多快"混为一谈。会快速估算数量级（采样率 × 位宽 vs 总线带宽），是嵌入式工程师的基本功，也是当场消除甲方顾虑、建立信任的利器。
<strong>你能用</strong>：遇到"会不会太快跟不上"的担心，立刻算"数据吞吐 = 采样率 × 位宽"，再对比总线/外设的额定速率，数量级一摆，焦虑自解。

### 5. 物料清单里的几个"经典件"
顺带出现的硬件，都是嵌入式里的常客，值得各自存档：
- <strong>STM32F412</strong>：主频约 100MHz 的 Cortex-M4 MCU（甲方一度口误说 M412/F412，对应 STM32F412 系列）；甲方提醒它"跟 FPGA 不一样"——原方案是 FPGA 做的，现在要移植到单片机。
- <strong>DS18B20</strong>：单总线数字温度探头，"经典的东西，很多"——温度采集的默认选型。
- <strong>独立 ADC 芯片（SPI 接口、24bit）</strong>：高精度采集场景常用外置 ADC 而非 MCU 内置（内置精度/速度不够）。
- <strong>加速度计</strong>：被列为"不关键"——如果"碰触/上电触发"那条路能工作，加速度计就不用了。
<strong>为什么重要</strong>：这套件几乎是"GPS 授时数采节点"的标准 BOM，认识它们就认识了一整类产品。
<strong>你能用</strong>：做类似设备时，把这份清单当起点选型，再按精度/功耗增删。

## 🛠 可直接复用的方法（含"不要误读成"）

<strong>方法 1：先确认能做，再谈价</strong>
- 核心意思：把需求澄清到双方都点头，再进入报价/合同；老手"先确定需求，不先说价格"。
- 你可以怎么用：接单第一通电话只干一件事——把需求和边界钉死（哪些你做、哪些甲方回填、交付时间）。
- 不要误读成：拖着不报价摆架子——是"需求没清不报价"，需求一清就要快（本场甲方"明天中午前定"）。

<strong>方法 2：用数量级估算消解对方的技术焦虑</strong>
- 核心意思：对方担心"快/跟不上"时，当场算吞吐 vs 带宽，用数量级差距让顾虑不攻自破。
- 你可以怎么用：备好"采样率 × 位宽 = 吞吐"和常见总线速率，随时口算。
- 不要误读成：嘴上说"没问题"——要给出可验证的数字，否则不建立信任。

<strong>方法 3：把"不用我做的部分"显式留接口</strong>
- 核心意思：老手主动说"处理那段东西不用你写，但把这个口子给我留出来"——明确划分外包边界。
- 你可以怎么用：接框架类外包时，把甲方自填的算法/业务逻辑约定成清晰接口（函数桩/数据缓冲），写进需求。
- 不要误读成：边界靠默契——要白纸黑字，否则交付时扯皮。
