# 脱敏门禁（公开发布前必过）

任何拆解产物进入公开仓库、GitHub Pages 或其他公域渠道之前，必须完整走完本门禁。**逐字稿原文、未脱敏的私有复盘，永不进入公开仓库。**

## 1. 发布前：保密请求扫描

在逐字稿与全部源材料中 grep 保密请求模式，命中即触发脱敏评审（该会议默认不公开，除非逐条确认可脱敏）：

```bash
grep -rniE "不要录音|别录音|不要外传|别发公域|不要发到|内部分享|内部交流|不要公开|off the record|do not record|confidential|internal only" <逐字稿或素材目录>
```

- 命中任何一条 → 该会议内容默认仅出私有版；若要出公开版，必须逐条确认脱敏后可公开，并在 `⚠️ 来源边界` 注明。
- 逐字稿原文本身**永不**进公开仓库——即使未命中保密模式，公开仓库也只放脱敏后的复盘产物。

## 2. 脱敏映射表（私有，绝不提交）

约定放在用户私有状态目录：`~/.miaoji-decon/redactions.json`。**此文件含真实实体名，绝不提交到公开仓库。**

格式为 JSON 映射对象：

```json
{
  "星辰科技": "某硬件公司",
  "张示例": "Speaker 3",
  "某集团客户实名": "某集团客户"
}
```

（以上为虚构示例，仅示意格式。）

规则：

- 同一会话、同一会议跨多篇报告**复用同一映射**，保证匿名一致——同一个真实实体在所有公开产物里替换成同一个匿名占位。
- 匿名占位要明显虚构、不可反推：用「某硬件公司」「某集团客户」「Speaker 3」这类，不用缩写、谐音或首字母。
- 新会议出现新实体时，先补进映射表再脱敏，不要临时发挥。

## 3. 脱敏范围指引

**必须脱敏的：**

- 合作方、客户、雇主的公司名与品牌名。
- 可定位到具体个人的组合信息：公司 + 职位 + 数字（如"某公司销售总监签了某金额合同"组合起来可定位，就要拆开匿名）。
- 灰色/敏感业务的可定位细节（具体打法 + 具体主体 + 具体数字的组合）。

**通常保留的：**

- 教训、方法论与数字本身一般有教学价值，予以保留；去掉的是**可定位性**，不是信息。
- 判断标准：读者能否据此定位到具体的人或公司。能定位 → 脱敏；不能定位 → 保留。

## 4. 发布后验证：敏感词 0 命中

公开产物（Markdown、HTML、索引页）生成后、发布前，对全部敏感词做 0 命中校验：

```bash
# 以 redactions.json 的键为词表，逐个 grep 公开产物，要求全部 0 命中
grep -rnF "星辰科技" docs/public/ <其他公开产物路径>
```

更省事的方式是用校验脚本的 `--banned-terms` 联动：

```bash
python3 skills/miaoji-decon/scripts/validate_deconstruction.py <公开产物.md> \
  --mode standard --banned-terms ~/.miaoji-decon/redactions.json
```

`--banned-terms` 接受字符串数组 JSON，也接受 `redactions.json` 这种映射对象（取其键作为敏感词表）；任一词命中即报错退出。

验证通过后，在报告的 `⚠️ 来源边界` 章节注明"公开版已脱敏"。

- [ ] 公开产物中无可定位到个人/公司的组合信息。
- [ ] `grep` 全部敏感词 0 命中 / `--banned-terms` 校验通过。
