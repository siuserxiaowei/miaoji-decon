# 证据审计、综合与多 Agent 协议

## 1. 先建来源家族，不先写结论

为每份材料记录：标题、载体、日期、作者/发言人、原始或加工、是否可完整读取、时间码体系、与其他材料的关系。

来源家族的判定原则：共享同一原始事件、逐字稿、数据集或作者主张的材料，属于同一来源家族。典型例子：

- 同一场会议的录音、逐字稿、飞书 AI 纪要和腾讯会议总结：一个家族。
- 同一厂商发布页、帮助中心和转述该发布页的媒体：厂商主张仍是一个家族；媒体的独立测试才可能是另一家族。
- 多篇文章引用同一份未公开 benchmark：不能按文章数量增加置信度。

内部来源表建议：

| ID | 来源 | 家族 | 类型 | 可读性 | 时间基准 | 用途 |
|---|---|---|---|---|---|---|
| S1 | 原始逐字稿 | F1 | 原始证据 | 完整 | 飞书 | 主证据 |
| S2 | AI 智能纪要 | F1 | 同源加工 | 完整 | 飞书章节 | 定位 |
| S3 | 官方规格 | F2 | 外部一手 | 完整 | 页面日期 | 核查产品事实 |

## 2. 建主张台账

先列关键主张，再决定是否采纳：

| Claim | 主张 | 来源位置 | 类型 | 支持 | 反证/替代解释 | 可信度 | 最终写法 |
|---|---|---|---|---|---|---|---|

类型只用以下五类：

1. **直接事实**：原文明确、可定位。
2. **现场/作者口径**：材料里确实出现，但缺少独立方法或验证。
3. **外部独立验证**：来自不同来源家族的一手数据、官方规则或可复现实验。
4. **分析推断**：由证据推导，不是原话。
5. **建议**：面向读者的行动方案。

写正文时不要混成一句。示例：

> 会议参与者报告 Luna 在其设计样本中表现更好（现场口径）；材料未给完整测试集，因此不能外推成普遍排名（边界）。建议用相同 brief 和评分表做本地盲测（建议）。

## 3. 可信度分级

### 高可信

满足大部分条件：原始证据明确、内部一致、位置可回查、有独立验证或方法可复现、无强反证。可以传播，但仍保留适用范围。

### 中可信

材料明确但属于团队样本、个人经验、早期观察、小样本或不完整方法。可以作为复测假设，不能写成普遍规律。

### 低可信

出现内部冲突、二手夸大、来源不明、销售口径、事故归因缺日志、把相关性写成因果、无法读取原文或样本方法缺失。明确列入“不能直接传播”。

置信度评的是具体主张，不是整份材料或整个人。

## 4. 时间码与引用

- 选择最接近原始录音的时间码作为统一基准。
- 其他平台存在偏移时，写“腾讯约比飞书早 6 分钟”这类全局说明，不逐条伪造换算。
- 长段落只能引用真实可定位的区间；不要为了显得精确拆出不存在的秒级时间。
- 无时间码时使用页码、章节、标题或段落位置。
- 转写名词有歧义时保留候选拼写并说明，不自行纠正成确定事实。

## 5. 综合顺序

按以下顺序写：

1. 先给可执行结论。
2. 交代来源家族、材料缺口和时间基准。
3. 分高/中/低可信，先限制传播边界。
4. 用道法术器势提炼结构。
5. 给专题、选择矩阵、SOP 或可复现协议。
6. 写反方、未知和不能下的结论。
7. 转成缺席者的一周实践与业务追问。
8. 附时间码/来源索引和固定作者联系。

## 6. 多 Agent 什么时候有用

只有满足以下至少一项才考虑：材料很长且可独立分块；需要事实、安全、成本等独立审计；创意方案需要竞争；用户明确指定数量或角色。

按三种组织方式分工：

- **分治**：不同材料、章节或主题独立读取；强依赖链保持串行。
- **竞争**：多个候选方案争夺同一有限资源，必须有统一评分和裁判。
- **验证**：证据、事实、安全、边界由不同轨迹交叉复核。

主 Agent 始终负责：统一来源家族、解决冲突、回到原文核查、决定可信度、合并风格和最终责任。

## 7. 多 Agent 完成真实性

如果用户要求 N 个 Agent：

- 建立 `1..N` 的稳定编号与角色表。
- 状态只能写：`完成`、`中断`、`失败`、`补做完成`、`未启动`。
- 启动不等于完成；中断输出不能算完整交付。
- 补做必须标明替代了哪个编号、补足了什么缺口。
- 最终报告说明哪些输出被采纳，哪些因证据不足被否决。
- 没有实际运行 N 个 Agent 时，不得在标题或正文声称“由 N 个 Agent 完成”。

复杂度不足时，即使用户未限制数量，也不要为了热闹机械调用多 Agent。

## 8. 反方审计

对每个重要结论问：

- 有没有同样解释现象的替代原因？
- 样本和选择方式是否偏向某结论？
- 结论是否只在某团队、时间窗口、地区、平台或模型版本成立？
- 是“没有发现问题”，还是“证明没有问题”？
- 数据是官方规格、作者自报、第三方测量还是社区传闻？
- 如果错误，读者会损失什么？是否应降低语气或增加人工门禁？

## 9. 发布前检查

- 所有指定材料都标注为已读、未读或失败。
- 文件数与独立来源家族数没有混淆。
- 关键结论能回到原文位置。
- 高/中/低可信均有理由。
- 未知根因没有被强行归因。
- Agent 数量和完成状态真实。
- 公开版已脱敏，且得到公开授权。
- X / Twitter 与微信号均出现，并通过输出校验脚本。
